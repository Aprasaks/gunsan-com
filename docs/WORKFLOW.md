# 군산.com 이슈·커밋·푸시 작업 기준 (WORKFLOW.md)

## 목적

이 문서는 군산.com 프로젝트의 작업 흐름을 정리합니다.

군산.com은 빠르게 만들 수 있지만, 허술하게 만들면 안 됩니다.
따라서 모든 작업은 작은 이슈 단위로 나누고, 하나의 커밋에는 하나의 의도만 담습니다.

---

## 기본 원칙

- 하나의 이슈는 하나의 작업 단위입니다.
- 하나의 커밋은 하나의 의도만 가집니다.
- 하나의 커밋에 여러 기능을 섞지 않습니다.
- 코드 변경 전에는 반드시 어떤 GitHub Issue에 해당하는 작업인지 확인합니다.
- 이슈 없는 큰 작업은 금지합니다.
- 문서 작업, UI 작업, 데이터 타입 작업, 더미 데이터 작업, Supabase 작업, 지도 작업, 사진 업로드 작업, 관리자 작업은 서로 다른 이슈로 분리합니다.
- 사용자가 요청하지 않은 기능을 미리 구현하지 않습니다.
- v0.1 범위 밖의 기능은 선제 구현하지 않습니다.

---

## Bootstrap Exception

프로젝트 초기 세팅 단계의 커밋은 GitHub Issue 없이 허용할 수 있습니다.

### 예외 범위

- #1 초기 Next.js 프로젝트 생성
- #2 프로젝트 작업 규칙 문서 추가
- #3 이슈 커밋 푸시 작업 기준 문서 추가

단, 이 예외는 초기 부트스트랩 단계에만 적용합니다.

**#4 이후의 기능, UI, 데이터, Supabase, 지도, 사진 업로드, 관리자 작업은 반드시 GitHub Issue를 먼저 생성한 뒤 진행합니다.**

가능하면 GitHub Issues를 소급 생성해 번호를 맞춥니다.

### 권장 소급 이슈

- Issue #1: 초기 Next.js 프로젝트 생성
- Issue #2: 프로젝트 작업 규칙 문서 추가
- Issue #3: 이슈 커밋 푸시 작업 기준 문서 추가

소급 생성한 이슈에는 다음과 같이 기록합니다.

```txt id="merrld"
초기 부트스트랩 단계에서 먼저 커밋한 작업입니다.
관련 커밋: <커밋 해시>
```

## 이슈 번호와 커밋 번호 기준

- 커밋 메시지의 `#번호`는 원칙적으로 GitHub Issue 번호를 의미합니다.
- 단, #1~#3은 초기 부트스트랩 예외로 허용합니다.
- **#4부터는 실제 GitHub Issue 번호와 커밋 메시지 번호를 반드시 맞춥니다.**
- GitHub Issue 번호와 커밋 메시지 번호를 맞출 수 없는 경우에는 작업을 시작하지 말고 사용자에게 확인합니다.
- 이슈 번호가 불명확하면 임의 번호를 붙이지 않습니다.

## 작업 범위 확인 기준

작업 시작 전 다음을 확인합니다.

1. 현재 작업이 어떤 GitHub Issue에 해당하는가?
2. 이번 이슈에서 수정할 파일은 무엇인가?
3. 이번 이슈에서 절대 수정하면 안 되는 파일은 무엇인가?
4. 이번 작업이 v0.1 범위를 벗어나지 않는가?
5. 사용자가 요청하지 않은 기능이 섞이지 않았는가?
6. 한 커밋에 여러 의도가 섞이지 않는가?

작업 범위가 불명확하면 사용자에게 확인합니다.

## 커밋 전 확인 기준

## 문서만 수정한 경우

문서만 수정한 경우에는 다음을 확인합니다.

```bash id="whp7yr"
git status
git diff
```

문서 작업에서는 `npm run build`를 생략할 수 있습니다.

단, 다음 조건을 만족해야 합니다.

- 변경 파일이 문서 파일만이어야 합니다.
- 코드 파일이 섞이면 안 됩니다.
- 문서 내용이 현재 프로젝트 방향과 충돌하지 않아야 합니다.

---

## 코드 수정이 있는 경우

코드 수정이 있는 경우에는 다음 순서로 확인합니다.

```bash id="ckcmwm"
npm run lint
npm run build
git status
git diff
```

조건:

- `npm run lint` 통과
- `npm run build` 통과
- 변경 범위가 이슈 범위 안에 있어야 함
- 요청 범위 밖 파일이 수정되지 않아야 함
- `.env` 등 민감한 파일이 포함되지 않아야 함

---

## lint 실행 불가 예외

프로젝트 초기 단계에서 `npm run lint` 스크립트가 아직 설정되지 않은 경우가 있을 수 있습니다.

이 경우 AI 에이전트는 commit/push를 즉시 중단하지 않고 다음 기준을 따릅니다.

- `npm run lint` 실행이 불가능하다는 사실을 보고합니다.
- `npm run build`가 통과하는지 확인합니다.
- 해당 이슈가 ESLint 설정 이전의 작업인지 확인합니다.
- 사용자가 명시적으로 `이슈커밋푸시 진행해`라고 지시한 경우, lint 실행 불가를 보고한 뒤 build 통과 상태에서 commit/push를 진행할 수 있습니다.
- 단, ESLint 설정 이슈가 완료된 이후에는 코드 변경 시 `npm run lint` 통과를 필수로 합니다.

### 주의

ESLint 설정과 lint 스크립트가 추가된 이후에는 이 예외를 남용하지 않습니다.

## 푸시 기준

main 브랜치에 push하기 전 다음 조건을 모두 충족해야 합니다.

- 작업이 하나의 이슈 범위 안에 있어야 합니다.
- 커밋 메시지에 이슈 번호가 있어야 합니다.
- 코드 변경이면 `npm run lint`와 `npm run build`가 통과해야 합니다.
- 문서 작업이면 변경 파일이 문서 파일만이어야 합니다.
- 실패한 빌드는 main에 push하지 않습니다.
- force push는 절대 하지 않습니다.

## 기본 작업 순서

일반적인 작업은 다음 순서로 진행합니다.

1. GitHub Issue 생성
2. 작업 범위 확인
3. 필요한 파일만 수정
4. 코드 변경이면 `npm run lint`
5. 코드 변경이면 `npm run build`
6. 문서 작업이면 `git diff` 확인
7. 변경 범위 확인
8. `git add`
9. `git commit -m "작업 내용 #이슈번호"`
10. `git push`
11. 이슈에 작업 결과 코멘트 작성
12. 이슈 닫기

## AI 에이전트 commit/push 권한 제한

AI 에이전트는 기본적으로 commit/push를 실행하지 않습니다.

AI 에이전트는 사용자의 명시적 지시 없이 다음을 실행하지 않습니다.

- `git commit`
- `git push`
- `git push --force`
- 배포 설정 변경
- 환경변수 변경
- 외부 서비스 연결
- Supabase 프로젝트 생성
- 도메인 설정 변경

기본적으로 AI는 다음까지만 수행합니다.

- 파일 읽기
- 코드 수정
- 문서 수정
- lint/build 실행
- 변경 사항 요약
- 제안 커밋 메시지 작성

## 이슈커밋푸시 자동 진행 규칙

기본적으로 AI 에이전트는 사용자의 명시적 지시 없이 commit/push를 실행하지 않습니다.

단, 사용자가 다음과 같이 명시하면 AI 에이전트는 이슈 생성, 커밋, 푸시까지 진행할 수 있습니다.

### 명령 예시

```txt id="zm6yfz"
이슈커밋푸시 진행해
이 작업 이슈 만들고 커밋하고 푸시까지 해줘
issue commit push까지 진행해
```

이 명령을 받은 경우 AI 에이전트는 해당 작업 범위 안에서만 commit/push 권한을 가집니다.

다음 작업에서 다시 commit/push가 필요하면 사용자가 다시 명시해야 합니다.

## 이슈커밋푸시 자동 진행 순서

## 1. 현재 변경 상태 확인

```bash id="dax6d8"
git status
git diff
```

확인할 것:

- 변경 파일 목록
- 변경 범위
- 의도하지 않은 파일 변경 여부
- `.env` 등 민감한 파일 포함 여부

---

## 2. 작업 범위 확인

변경 파일이 현재 작업 목적과 맞는지 확인합니다.

요청 범위 밖 파일이 섞여 있으면 중단하고 사용자에게 보고합니다.

---

## 3. GitHub Issue 생성

GitHub CLI 사용 가능 시 `gh issue create`로 이슈를 생성합니다.

이슈 제목은 작업 내용을 명확히 설명하는 한국어 문장으로 작성합니다.

이슈 본문에는 다음을 포함합니다.

- 목표
- 작업 범위
- 요구사항
- 제외 범위
- 완료 기준
- 확인 방법

`gh` CLI를 사용할 수 없으면 이슈 생성용 제목과 본문을 제안하고 중단합니다.

## 자동 진행 중단 조건

다음 상황에서는 commit/push를 중단하고 사용자에게 보고합니다.

## CI/CD 도구 기준

현재 v0.1 단계에서는 Harness 같은 별도 CI/CD 도구를 사용하지 않습니다.

### 이유

- 아직 MVP 초기 단계입니다.
- GitHub Issue, 작은 커밋, `npm run lint`, `npm run build`, Vercel Preview 확인만으로 충분합니다.
- 자동화 도구를 붙이기 전에 작업 단위와 품질 기준이 먼저 필요합니다.

추후 필요하면 **GitHub Actions**부터 검토합니다.

Harness나 고급 CI/CD 도구는 팀 규모가 커지거나 배포 파이프라인이 복잡해진 뒤 검토합니다.

## 최종 기준

군산.com 작업은 빠르게 진행하되, 기준 없이 밀어붙이지 않습니다.

- 생각은 크게 합니다.
- 구현은 작게 합니다.
- 이슈는 잘게 나눕니다.
- 커밋은 하나의 의도만 담습니다.
- 검증되지 않은 정보는 검증된 것처럼 보이게 하지 않습니다.
- AI는 사용자의 명시적 권한 없이 commit/push하지 않습니다.

군산.com의 큰 비전은 **“군산의 모든 것”** 입니다.
하지만 개발은 반드시 작은 작업 단위로 진행합니다.
