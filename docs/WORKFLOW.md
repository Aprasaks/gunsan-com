# 군산.com 이슈·커밋·푸시 작업 기준

## 목적

군산.com은 장소 포털에서 코스 중심 여행 큐레이션 플랫폼으로 전환합니다. 방향 전환은 한 번에 전체 코드를 바꾸는 작업이 아닙니다. 문서, 타입, seed, UI, API를 작은 이슈로 나누고 하나의 커밋에는 하나의 의도만 담습니다.

## 기본 원칙

- 하나의 이슈는 하나의 작업 단위입니다.
- 하나의 커밋은 하나의 의도만 가집니다.
- 문서, 타입, 데이터, UI, API, asset, 배포 작업을 섞지 않습니다.
- 작업 전 실제 GitHub Issue 번호와 범위를 확인합니다.
- 실제 이슈 번호가 없거나 불명확하면 commit/push하지 않습니다.
- 사용자 지시 없이 commit/push하지 않습니다.
- force push하지 않습니다.
- 기존 미커밋 변경은 사용자 작업으로 보고 보존합니다.

## 코스 중심 전환 작업 분리

권장 순서:

1. 방향 전환 문서 정리
2. Course / CourseStop 타입 설계
3. 초기 Course seed 데이터
4. 군산 필수 5 콘텐츠
5. 대표 먹거리 5 콘텐츠
6. 군산다운 카페 5 콘텐츠
7. 메인 코스 선택 UI
8. 코스 목록/상세 페이지
9. Place 상세의 코스 맥락 보강
10. 공식 출처 검증 API 단일 연결

다음 조합은 하나의 이슈로 진행하지 않습니다.

- 문서 전환 + Course 타입 구현
- Course 타입 + seed 데이터
- seed 데이터 + 메인 UI
- 메인 UI + 지도 SDK
- 코스 콘텐츠 + 행사 API
- API 연결 + 대규모 디자인 변경

## 현재 작업 단계

현재 최상위 방향은 다음과 같습니다.

> 군산을 처음 온 사람이 그대로 따라갈 수 있는 코스와 필수 선택지를 정해주는 여행 큐레이션 플랫폼

“군산의 모든 것”은 장기 데이터 확장 방향이며 초기 메인과 작업 우선순위의 기준이 아닙니다.

현재 우선 작업 후보:

- Course 도메인 설계
- 군산 처음 반나절 / 1박 2일 / 2박 3일 코스
- 아이랑 / 비 오는 날 / 선유도 코스
- 군산 필수 5
- 대표 먹거리 5
- 군산다운 카페 5
- 코스 선택 중심 메인

생활업체, 병원/약국, 네일/뷰티 등 생활 정보 확장은 후순위이며 코스 전환 작업과 섞지 않습니다.

## 이슈 작성 기준

### 제목

끝낼 결과가 한 문장으로 드러나야 합니다.

좋은 예:

- 군산.com 코스 중심 방향 전환 문서 정리
- Course와 CourseStop 타입 설계
- 군산 처음 반나절 코스 seed 작성
- 군산 필수 5 콘텐츠 초안 정리
- 메인 코스 선택 영역 구현
- 코스 상세 타임라인 구현
- 행사 공식 출처 검증 기준 정리

나쁜 예:

- 군산.com 전체 전환
- 코스 전부 구현
- 타입 데이터 UI 한 번에 수정
- 메인 거의 완성
- 여러 가지 업데이트

### 이슈 본문

```markdown
## 목표

이번 이슈에서 끝낼 결과를 한 문장으로 작성합니다.

## 작업 범위

- 수정하거나 생성할 파일

## 요구사항

- [ ] 필수 요구사항

## 제외 범위

- 이번 이슈에서 하지 않을 작업

## 완료 기준

- 검증 가능한 완료 조건

## 확인 방법

- 실행할 명령과 확인할 경로
```

## 작업 유형별 경계

### 문서 작업

예:

- `AGENTS.md`
- `CLAUDE.md`
- `docs/DESIGN.md`
- `docs/SKILL.md`
- `docs/WORKFLOW.md`
- API 조사 문서

문서 작업에서는 코드, 타입, seed, asset을 수정하지 않습니다.

확인:

```bash
git status --short
git diff -- '*.md'
git diff --check
```

문서만 수정하면 `npm run build`는 생략할 수 있습니다.

### 타입 작업

Course와 CourseStop의 식별자, union, optional 여부를 정합니다.

- seed 데이터 작성 금지
- UI 구현 금지
- API 연결 금지
- 기존 Place 타입을 무관하게 대규모 변경 금지

코드 변경이므로 lint와 build를 실행합니다.

### seed 데이터 작업

한 이슈에서 한 코스 또는 하나의 콘텐츠 묶음만 다룹니다.

예:

- 군산 처음 반나절 코스
- 군산 필수 5
- 대표 먹거리 5

검증되지 않은 날짜, 거리, 시간, 가격을 만들지 않습니다. 공식 출처, 마지막 확인일, 편집 근거를 기록합니다.

### UI 작업

컴포넌트 또는 화면 한 단위로 진행합니다.

예:

- 메인 코스 선택 영역
- CourseCard
- CourseStop 타임라인
- 모바일 첫 화면 압축

UI 작업에 타입 대규모 변경, seed 작성, API 연결을 섞지 않습니다.

### 콘텐츠 작업

콘텐츠 작업은 맛 평가가 아니라 여행 맥락을 정리합니다.

- 왜 가는가
- 어느 코스에 들어가는가
- 얼마나 머무는가
- 다음 장소와 대체 장소
- 주차, 아이, 비 오는 날 조건
- 공식 출처와 마지막 확인일

외부 리뷰와 본문을 복사하지 않습니다.

### API 작업

API 하나를 하나의 이슈로 연결합니다.

API 목적은 장소 수 확대가 아니라 코스/필수 장소 검증 보강입니다.

- `lib/` 또는 `lib/api/`에 호출 로직 배치
- 외부 응답은 `unknown`에서 검증
- 로딩, 실패, fallback 처리
- 공식 출처와 갱신 기준 기록
- 가짜 실시간 상태 금지
- API 데이터로 추천 순위를 자동 생성하지 않음

### 지도 작업

지도는 코스 이동을 이해하는 보조 기능입니다.

- SDK 연결
- 좌표 검증
- 코스 선/마커
- 모바일 상호작용

위 항목은 필요하면 더 작은 이슈로 나눕니다. 실제 지도 연결 전 허위 지도 컨트롤을 만들지 않습니다.

### asset 작업

- 직접 제작, 직접 촬영, 제공받은 이미지, 사용권한 확인 이미지 사용
- 출처, 사용 위치, 용량, alt 기록
- 외부 리뷰/지도/업체 사진 무단 저장 금지
- 실제 군산 장소로 오해되는 허위 생성 이미지 금지

### 배포 작업

기능 구현과 배포를 분리합니다. 환경변수, 도메인, 외부 서비스 변경은 사용자 승인을 받습니다.

## 작업 전 체크리스트

1. 실제 Issue 번호가 있는가?
2. 이번 작업 유형은 문서, 타입, 데이터, UI, API 중 무엇인가?
3. 수정 파일과 수정 금지 파일이 명확한가?
4. Course 중심 방향과 맞는가?
5. Place 목록을 늘리는 작업으로 변질되지 않았는가?
6. 검증되지 않은 수치, 날짜, 시간, 순위를 만들지 않았는가?
7. 공식 정보, 편집 추천, 업체 제공, 방문자 확인을 구분했는가?
8. 외부 콘텐츠 사용 권한이 확인되었는가?
9. 기존 미커밋 변경과 겹치지 않는가?

## 코드 검증

코드 변경 시:

```bash
npm run lint
npm run build
git status --short
git diff
git diff --check
```

조건:

- lint 통과
- build 통과
- TypeScript `any` 없음
- 요청 범위 밖 변경 없음
- 민감한 파일 없음
- 의도하지 않은 lockfile 변경 없음

## 문서 검증

문서만 변경 시:

```bash
git status --short
git diff -- '*.md'
git diff --check
```

확인할 내용:

- 코드 파일을 이번 작업에서 수정하지 않았는가?
- 문서 간 최상위 브랜드가 일치하는가?
- Course가 Place보다 중심으로 정의되는가?
- 생활 정보가 초기 메인 우선순위에서 내려갔는가?
- 검증과 출처 원칙이 유지되는가?

작업 전부터 존재한 코드 변경은 되돌리지 않고 별도로 보고합니다.

## 커밋 메시지

형식:

```txt
작업 내용 #실제이슈번호
```

예:

```txt
군산.com 코스 중심 방향 전환 문서 정리 #42
Course와 CourseStop 타입 설계 #43
군산 처음 반나절 코스 seed 작성 #44
```

실제 Issue 번호가 없으면 임의 번호를 붙이지 않고 commit을 중단합니다.

## AI 권한

사용자의 명시적 지시 없이 다음을 실행하지 않습니다.

- `git commit`
- `git push`
- 배포
- 환경변수 변경
- 외부 서비스 연결
- Issue 생성
- 유료 서비스 설정

사용자가 “이슈커밋푸시 진행해”라고 명시하더라도 실제 Issue 번호, 검증 결과, 변경 범위가 확인되어야 합니다.

## 브랜치

다음 작업은 별도 브랜치를 권장합니다.

- Course 도메인 대규모 도입
- 코스 목록/상세 라우트 추가
- 지도 SDK
- Supabase/DB
- 로그인/권한
- 결제
- 대규모 레이아웃 전환

예:

```txt
feature/issue-43-course-types
feature/issue-44-first-course-seed
feature/issue-50-course-home
```

- ...

## 금지

- 이슈 없이 큰 기능 구현
- 한 커밋에 문서와 코드 혼합
- Course 타입, seed, UI 동시 구현
- 실제 번호가 아닌 대화용 번호를 커밋에 사용
- 실패한 build를 push
- force push
- 외부 콘텐츠 무단 복사/크롤링
- 광고를 편집 추천으로 위장
- “모든 장소를 많이 보여주는 것”을 제품 목표로 되돌리기
