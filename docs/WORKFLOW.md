# 군산.com 이슈·커밋·푸시 작업 기준 (WORKFLOW.md)

---

## 기본 원칙

- 하나의 이슈는 하나의 작업 단위다.
- 하나의 커밋은 하나의 의도만 가진다.
- 하나의 커밋에 여러 기능을 섞지 않는다.
- 코드 변경 전에는 반드시 어떤 이슈에 해당하는 작업인지 확인한다.
- 이슈 없는 큰 작업은 금지한다.
- 문서 작업, UI 작업, 데이터 타입 작업, Supabase 작업, 지도 작업, 사진 업로드 작업, 관리자 작업은 서로 다른 이슈로 분리한다.

---

## Bootstrap Exception

프로젝트 초기 세팅 단계의 커밋은 GitHub Issue 없이 허용할 수 있다.

**예외 범위:**

- #1 초기 Next.js 프로젝트 생성
- #2 프로젝트 작업 규칙 문서 추가
- #3 이슈 커밋 푸시 작업 기준 문서 추가

단, 이 예외는 초기 부트스트랩 단계에만 적용한다.
**#4 이후의 기능, UI, 데이터, Supabase, 지도, 사진 업로드, 관리자 작업은 반드시 GitHub Issue를 먼저 생성한 뒤 진행한다.**

가능하면 GitHub Issues를 소급 생성해 번호를 맞춘다.

**권장 소급 이슈:**

- Issue #1: 초기 Next.js 프로젝트 생성
- Issue #2: 프로젝트 작업 규칙 문서 추가
- Issue #3: 이슈 커밋 푸시 작업 기준 문서 추가

소급 생성한 이슈에는 "초기 부트스트랩 단계에서 먼저 커밋한 작업"이라고 기록한다.

---

## 이슈 번호와 커밋 번호 기준

- 커밋 메시지의 `#번호`는 원칙적으로 GitHub Issue 번호를 의미한다.
- 단, #1~#3은 초기 부트스트랩 예외로 허용한다.
- **#4부터는 실제 GitHub Issue 번호와 커밋 메시지 번호를 반드시 맞춘다.**
- GitHub Issue 번호와 맞출 수 없는 경우에는 작업을 시작하지 말고 사용자에게 확인한다.

## 이슈 작성 기준

### 제목 형식

간결하고 명확하게. 무엇을 끝내는지 한 줄로 표현한다.

**좋은 예:**

- 프로젝트 작업 규칙 문서 추가
- 사이트 메타데이터와 전역 스타일 정리
- 장소 타입과 더미 데이터 추가
- 검증 상태 배지 컴포넌트 구현
- 장소 카드 컴포넌트 구현
- 메인 페이지 조립
- 제보하기 페이지 구현
- Supabase 클라이언트 설정 추가

### Issue body 형식

```markdown
## 목표

이번 이슈에서 끝내야 하는 목표를 한 문장으로 작성한다.

## 작업 범위

수정하거나 생성할 파일 목록을 작성한다.

## 요구사항

- [ ] 구현해야 하는 요구사항 1
- [ ] 구현해야 하는 요구사항 2

## 제외 범위

이번 이슈에서 절대 하지 말아야 할 작업을 작성한다.

## 완료 기준

이슈를 닫을 수 있는 조건을 작성한다.

## 확인 방법

로컬에서 확인할 명령어와 페이지 경로를 작성한다.
```

## 이슈 크기 기준

### 좋은 이슈 (하나의 이슈가 담아야 할 범위)

- 하나의 컴포넌트 구현
- 하나의 페이지 구현
- 하나의 타입 파일 추가
- 하나의 문서 추가
- 하나의 DB 연결 작업
- 하나의 폼 저장 작업

### 나쁜 이슈 (이런 이슈는 쪼개야 한다)

- 군산.com 전체 구현
- 메인, 목록, 상세, 제보, 사장님 페이지 한 번에 구현
- Supabase와 지도와 사진 업로드를 한 번에 구현
- UI 리팩토링과 데이터 구조 변경을 한 번에 수행
- 디자인 전체 개선

## 커밋 메시지 기준

커밋 메시지는 **한국어**로 작성한다.

**형식:**
```

```

**좋은 예:**

- `프로젝트 작업 규칙 문서 추가 #2` (부트스트랩 예외)
- `이슈 커밋 푸시 작업 기준 문서 추가 #3` (부트스트랩 예외)
- `사이트 메타데이터와 전역 스타일 정리 #4`
- `장소 타입과 더미 데이터 추가 #5`
- `검증 상태 배지 컴포넌트 구현 #6`
- `장소 카드 컴포넌트 구현 #7`

**나쁜 예:**

- `수정`
- `작업함`
- `update`
- `ui`
- `여러가지 수정`
- `메인 거의 완성`

## 커밋 전 확인 기준

### 문서만 수정한 경우

- `git diff` 확인
- markdown 내용 확인
- `npm run build` 생략 가능

### 코드 수정이 있는 경우

1. `npm run lint`
2. `npm run build`
3. `git diff` 확인
4. 변경 범위가 이슈 범위 안에 있는지 확인

## 푸시 기준

main 브랜치에 push하기 전 다음 조건을 모두 충족해야 한다.

- 작업이 하나의 이슈 범위 안에 있어야 한다.
- 커밋 메시지에 이슈 번호가 있어야 한다.
- 코드 변경이면 `npm run lint`와 `npm run build`가 통과해야 한다.
- 실패한 빌드는 main에 push하지 않는다.
- 문서 작업은 build 없이 push 가능하지만, 변경 파일이 문서 파일만이어야 한다.

- 배포는 기능 구현과 별도 이슈로 분리한다.
- 배포 관련 커밋도 하나의 이슈 범위 안에서 처리한다.

## AI 에이전트 commit/push 권한 제한

- AI 에이전트는 **사용자가 명시적으로 지시하지 않으면 commit을 실행하지 않는다.**
- AI 에이전트는 **사용자가 명시적으로 지시하지 않으면 push를 실행하지 않는다.**
- 기본적으로 AI는 변경 파일, 주요 변경, 확인 결과, 제안 커밋 메시지만 보고한다.
- **commit/push는 사용자가 직접 실행하는 것을 기본으로 한다.**
- 사용자가 "이슈커밋푸시 진행해" 등으로 명시한 경우에 한해 AI가 이슈 생성, commit, push를 순서대로 실행한다. 자세한 규칙은 아래 "이슈커밋푸시 자동 진행 규칙" 참조.

---

## 이슈커밋푸시 자동 진행 규칙

기본적으로 AI 에이전트는 사용자의 명시적 지시 없이 commit/push를 실행하지 않는다.

단, 사용자가 다음과 같이 명시하면 AI 에이전트는 이슈 생성, 커밋, 푸시까지 진행할 수 있다.

**명령 예시:**

- `이슈커밋푸시 진행해`
- `이 작업 이슈 만들고 커밋하고 푸시까지 해줘`
- `issue commit push까지 진행해`

이 명령을 받은 경우 AI 에이전트는 다음 순서를 따른다.

**1. 현재 변경 상태 확인**

- `git status`
- `git diff`

**2. 작업 범위 확인**

- 변경 파일이 현재 작업 목적과 맞는지 확인한다.
- 요청 범위 밖 파일이 섞여 있으면 중단하고 사용자에게 보고한다.

**3. GitHub Issue 생성**

- GitHub CLI 사용 가능 시 `gh issue create`로 이슈를 생성한다.
- 이슈 제목은 작업 내용을 명확히 설명하는 한국어 문장으로 작성한다.
- 이슈 본문에는 목표, 작업 범위, 요구사항, 제외 범위, 완료 기준, 확인 방법을 포함한다.
- `gh` CLI를 사용할 수 없으면 이슈 생성용 제목과 본문을 제안하고 중단한다.

**4. 검증**

- 문서만 수정한 경우: `git diff`로 변경 범위만 확인한다.
- 코드 변경이 있는 경우: `npm run lint` → `npm run build` 순서로 실행한다.
- lint 또는 build 실패 시 commit/push하지 않고 사용자에게 보고한다.

**5. 커밋**

- 커밋 메시지는 `작업 내용 #이슈번호` 형식으로 작성한다.
- 예: `장소 타입과 더미 데이터 추가 #5`
- 하나의 커밋에는 하나의 작업만 포함한다.

**6. 푸시**

- v0.1 작은 작업은 main에 직접 push할 수 있다.
- Supabase, 지도, 사진 업로드, 관리자 페이지, 대규모 레이아웃 변경은 별도 브랜치를 사용한다.
- force push는 절대 하지 않는다.

**7. 작업 보고**

완료 후 다음 항목을 보고한다.

- 생성한 이슈 번호
- 커밋 해시
- push한 브랜치
- 실행한 검증 명령
- 남은 TODO

## 자동 진행 중단 조건

다음 상황에서는 commit/push를 중단하고 사용자에게 보고한다.

- 작업 범위 밖 파일이 포함됨
- `.env` 또는 비밀 키 파일이 포함됨
- `package-lock.json` 등 의도하지 않은 대규모 변경 발생
- `npm run lint` 실패
- `npm run build` 실패
- 이슈 번호 생성 실패
- 커밋 메시지에 이슈 번호를 붙일 수 없음
- 사용자가 요청하지 않은 기능이 섞임
- v0.1 범위 밖 기능이 몰래 추가됨

## 이슈커밋푸시 권한 기준

"이슈커밋푸시 진행해" 명령은 **해당 작업에 한해서만** commit/push 권한을 허용한다.

다음 작업에서 다시 commit/push가 필요하면 사용자가 다시 명시해야 한다.

## 절대 금지

- 이슈 없이 큰 기능 구현
- 한 커밋에 여러 기능 섞기
- TypeScript `any` 사용
- 요청 범위 밖 파일 수정
- v0.1에서 Supabase, 지도, 사진 업로드, 관리자 페이지를 몰래 추가
- 업체 제공 정보와 방문자 확인 정보 섞기
- 확인 필요 정보를 "최근 확인됨"으로 표시
- 실패한 build 상태로 main push
- **사용자 지시 없이 AI가 commit 실행**
- **사용자 지시 없이 AI가 push 실행**

## CI/CD 도구 기준

현재 v0.1 단계에서는 Harness 같은 별도 CI/CD 도구를 사용하지 않는다.

**이유:**

- 아직 MVP 초기 단계다.
- GitHub Issue, 작은 커밋, `npm run lint`, `npm run build`, Vercel 배포 확인만으로 충분하다.
- 자동화 도구를 붙이기 전에 작업 단위와 품질 기준이 먼저 필요하다.

추후 필요하면 **GitHub Actions**부터 검토한다.
Harness나 고급 CI/CD 도구는 팀 규모가 커지거나 배포 파이프라인이 복잡해진 뒤 검토한다.
