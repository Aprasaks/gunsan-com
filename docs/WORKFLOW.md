# 군산.com 이슈·커밋·푸시 작업 기준 (WORKFLOW.md)

## 목적

이 문서는 군산.com 프로젝트의 작업 흐름을 정리합니다.

군산.com은 빠르게 만들 수 있지만, 허술하게 만들면 안 됩니다.

따라서 모든 작업은 작은 작업 단위로 나누고, 하나의 커밋에는 하나의 의도만 담습니다.

군산.com은 현재 **레퍼런스형 포털 홈 구축 단계**입니다.

디자인, 문서, UI, 데이터, API, 배포 작업은 서로 섞지 않고 분리해서 진행합니다.

---

## 기본 원칙

- 하나의 이슈는 하나의 작업 단위입니다.
- 하나의 커밋은 하나의 의도만 가집니다.
- 하나의 커밋에 여러 기능을 섞지 않습니다.
- 코드 변경 전에는 현재 작업이 어떤 GitHub Issue에 해당하는지 확인합니다.
- 이슈 없는 큰 작업은 금지합니다.
- 작업 범위가 불명확하면 먼저 사용자에게 확인합니다.
- 사용자가 요청하지 않은 기능을 미리 구현하지 않습니다.
- 디자인 작업, 문서 작업, UI 작업, 데이터 타입 작업, API 작업, 배포 작업은 서로 다른 작업으로 분리합니다.
- Supabase, 지도, 사진 업로드, 관리자, 로그인, 결제, 대량 API 연결은 작은 UI 작업과 섞지 않습니다.
- 레퍼런스 이미지 기준 UI 보정도 섹션 단위로 나누어 진행합니다.

---

## 현재 작업 단계 기준

현재 군산.com은 초기 placeholder 사이트에서 벗어나, 사용자가 제공한 레퍼런스 이미지를 기준으로 포털형 홈을 구축하는 단계입니다.

현재 자주 발생하는 작업 유형은 다음과 같습니다.

- 문서 기준 정리
- Header 레퍼런스형 구조 보정
- 군산 현재 날씨 API 단일 연결
- SearchHero full-width 배너 보정
- HomeCategoryGrid 카드형 UI 보정
- 카테고리 이미지 적용
- PopularInfoSection chip UI 보정
- 군산 장소 둘러보기 preview 정리
- 장소 카드 상세페이지 연결
- TodayGunsanSection 보정
- EventFestivalSection 보정
- MapPreviewSection 보정
- NeighborhoodSection 보정
- `/places` 탐색 페이지 정리
- `/places/[slug]` 상세 페이지 강화
- `/submit` 제보 페이지 정리
- `/owners` 사장님 등록 안내 페이지 정리
- 반응형 보정
- Vercel Preview 배포

각 작업은 가능한 한 하나의 이슈와 하나의 커밋으로 분리합니다.

## Bootstrap Exception

프로젝트 초기 세팅 단계의 커밋은 GitHub Issue 없이 허용할 수 있습니다.

### 예외 범위

- #1 초기 Next.js 프로젝트 생성
- #2 프로젝트 작업 규칙 문서 추가
- #3 이슈 커밋 푸시 작업 기준 문서 추가

단, 이 예외는 초기 부트스트랩 단계에만 적용합니다.

**#4 이후의 기능, UI, 데이터, API, 문서, 배포 작업은 원칙적으로 GitHub Issue를 먼저 생성한 뒤 진행합니다.**

가능하면 GitHub Issues를 소급 생성해 번호를 맞춥니다.

### 권장 소급 이슈

- Issue #1: 초기 Next.js 프로젝트 생성
- Issue #2: 프로젝트 작업 규칙 문서 추가
- Issue #3: 이슈 커밋 푸시 작업 기준 문서 추가

소급 생성한 이슈에는 다음처럼 기록합니다.

```txt
초기 부트스트랩 단계에서 먼저 커밋한 작업입니다.
관련 커밋: <커밋 해시>
```

## 대화용 작업명과 GitHub Issue 번호 구분

ChatGPT, Claude, Codex와 대화하면서 부르는 작업명이나 임시 번호는 실제 GitHub Issue 번호가 아닐 수 있습니다.

예를 들어 대화에서 다음처럼 말할 수 있습니다.

```txt
작업명: 카테고리 카드 부제목 제거 및 정렬 보정
작업명: 자주 확인하는 군산 정보 칩 UI 보정
작업명: 군산 현재 날씨 API 연결
```

이 작업명은 작업 범위를 설명하기 위한 이름입니다.

커밋 메시지에 들어가는 `#번호`는 반드시 실제 GitHub Issue 번호여야 합니다.

### 원칙

- 대화용 작업 번호와 GitHub Issue 번호를 섞지 않습니다.
- 커밋 메시지의 `#번호`는 GitHub Issue 번호입니다.
- GitHub Issue 번호가 불명확하면 커밋 메시지에 임의 번호를 붙이지 않습니다.
- 사용자가 “작업명 기준으로 진행”한다고 말한 경우, 작업 보고는 작업명 기준으로 할 수 있습니다.
- 하지만 commit 전에는 실제 GitHub Issue 번호를 확인합니다.
- 실제 Issue 번호를 확인할 수 없으면 commit/push를 중단하고 사용자에게 묻습니다.

### 좋은 예

```txt
작업 보고 기준:
작업명: 자주 확인하는 군산 정보 칩 UI 보정

커밋 메시지:
자주 확인하는 군산 정보 칩 UI 보정 #27
```

### 나쁜 예

```txt
자주 확인하는 군산 정보 칩 UI 보정 #자주확인
카테고리 수정 #임시
메인 거의 완성 #55
```

## 이슈 번호와 커밋 번호 기준

- 커밋 메시지의 `#번호`는 원칙적으로 GitHub Issue 번호를 의미합니다.
- 단, #1~#3은 초기 부트스트랩 예외로 허용합니다.
- #4부터는 실제 GitHub Issue 번호와 커밋 메시지 번호를 맞춥니다.
- GitHub Issue 번호와 커밋 메시지 번호를 맞출 수 없는 경우에는 commit/push를 진행하지 않습니다.
- 이슈 번호가 불명확하면 임의 번호를 붙이지 않습니다.
- 커밋 메시지에 대화용 임시 번호를 사용하지 않습니다.

## 작업 범위 확인 기준

작업 시작 전 다음을 확인합니다.

1. 현재 작업이 어떤 GitHub Issue 또는 작업명에 해당하는가?
2. 이번 작업에서 수정할 파일은 무엇인가?
3. 이번 작업에서 절대 수정하면 안 되는 파일은 무엇인가?
4. 이번 작업이 현재 프로젝트 단계와 맞는가?
5. 사용자가 요청하지 않은 기능이 섞이지 않았는가?
6. 한 커밋에 여러 의도가 섞이지 않는가?
7. 실제 데이터처럼 보이는 가짜 수치나 날짜가 들어가지 않았는가?
8. “준비 중”, “정리 예정” 문구를 메인 화면에 반복 노출하지 않았는가?
9. 외부 사진, 리뷰, 지도 이미지를 무단으로 사용하지 않았는가?
10. TypeScript `any`를 사용하지 않았는가?

작업 범위가 불명확하면 사용자에게 확인합니다.

## 작업 유형별 분리 기준

## 문서 작업

문서 작업은 코드 작업과 섞지 않습니다.

예:

- `AGENTS.md` 수정
- `CLAUDE.md` 수정
- `docs/DESIGN.md` 수정
- `docs/SKILL.md` 수정
- `docs/WORKFLOW.md` 수정

문서 작업에서는 코드 파일을 수정하지 않습니다.

문서만 수정한 경우 `npm run build`는 생략할 수 있습니다.

---

## UI 작업

UI 작업은 한 섹션 또는 한 컴포넌트 단위로 분리합니다.

예:

- Header 보정
- SearchHero 보정
- HomeCategoryGrid 보정
- PopularInfoSection 보정
- TodayGunsanSection 보정
- EventFestivalSection 보정
- MapPreviewSection 보정
- NeighborhoodSection 보정

UI 작업 중 실제 API 연결, DB 저장, 페이지 생성, 타입 대규모 변경을 섞지 않습니다.

---

## API 작업

API 작업은 하나의 API만 다룹니다.

예:

- 군산 현재 날씨 API 연결

API 작업 기준:

- 하나의 API를 하나의 작업으로 분리합니다.
- API 호출 로직은 `lib/` 또는 `lib/api/`에 둡니다.
- 컴포넌트 안에 API 호출 코드를 길게 작성하지 않습니다.
- 로딩, 실패, fallback을 처리합니다.
- 실제 값이 없으면 가짜 수치를 만들지 않습니다.
- 여러 API를 한 번에 붙이지 않습니다.

---

## 데이터 작업

데이터 작업은 표시용 데이터와 실제 장소 데이터를 구분합니다.

예:

- `data/places.ts` 수정
- `data/homeCategories.ts` 추가
- `data/popularInfoItems.ts` 추가

주의:

- 홈 표시용 카테고리와 실제 `PlaceCategory`를 섞지 않습니다.
- 숙박, 관광/여행, 행사·축제를 실제 `PlaceCategory`에 임의로 추가하지 않습니다.
- 외부 데이터를 무단 복사하지 않습니다.
- 실제 확인하지 않은 정보를 확정처럼 쓰지 않습니다.

---

## asset 작업

이미지 asset 작업은 가능한 한 별도 이슈로 분리합니다.

허용:

- 사용자가 직접 제작한 이미지
- AI로 직접 생성한 이미지
- 사용 권한이 확인된 이미지
- 직접 촬영한 이미지
- WebP 등 최적화된 이미지

금지:

- 외부 사진 무단 다운로드
- 외부 리뷰 이미지 무단 사용
- 외부 지도 이미지 무단 사용
- 실제 군산 장소처럼 보이는 허위 AI 이미지
- 실제 상호, 간판, 건물을 허위로 생성한 이미지

asset 작업 보고에는 다음을 포함합니다.

- 추가한 파일 경로
- 파일 용량
- 사용 위치
- alt 문구
- 사용 권한 또는 제작 방식

## 배포 작업

배포 작업은 기능 구현과 분리합니다.

Vercel Preview 배포는 현재 단계에서도 가능합니다.

custom domain 연결도 별도 작업으로 시도할 수 있습니다.

단, 실제 운영 공개는 다음 기능이 붙은 뒤 검토합니다.

- 제보 저장
- 사진 업로드 또는 사진 제공 흐름
- 관리자 승인
- 최소한의 정보 검수 흐름

배포 작업에서는 환경변수, 도메인, 외부 서비스 연결 여부를 반드시 사용자에게 확인합니다.

## 커밋 메시지 기준

커밋 메시지는 한국어로 작성합니다.

### 형식

```txt
작업 내용 #이슈번호
```

### 좋은 예

```txt
프로젝트 작업 규칙 문서 추가 #2
이슈 커밋 푸시 작업 기준 문서 추가 #3
상단바 레퍼런스형 구조 보정 #18
군산 현재 날씨 API 연결 #19
카테고리 카드 부제목 제거 및 정렬 보정 #21
자주 확인하는 군산 정보 칩 UI 보정 #22
군산 장소 둘러보기 상세페이지 연결 #23
```

### 이슈 번호가 없을 때

GitHub Issue 번호가 확인되지 않으면 커밋 메시지에 임의 번호를 붙이지 않습니다.

이 경우 AI 에이전트는 커밋을 중단하고 사용자에게 확인합니다.

### 나쁜 예

```txt
수정
작업함
update
ui
여러가지 수정
메인 거의 완성
카테고리 수정 #임시
문서수정 #대충
```

## 커밋 전 확인 기준

## 문서만 수정한 경우

문서만 수정한 경우에는 다음을 확인합니다.

```bash
git status
git diff
```

문서 작업에서는 `npm run build`를 생략할 수 있습니다.

단, 다음 조건을 만족해야 합니다.

- 변경 파일이 문서 파일만이어야 합니다.
- 코드 파일이 섞이면 안 됩니다.
- 문서 내용이 현재 프로젝트 방향과 충돌하지 않아야 합니다.
- AGENTS, CLAUDE, DESIGN, SKILL, WORKFLOW 사이의 기준이 서로 충돌하지 않아야 합니다.

---

## 코드 수정이 있는 경우

코드 수정이 있는 경우에는 다음 순서로 확인합니다.

```bash
npm run lint
npm run build
git status
git diff
```

조건:

- `npm run lint` 통과
- `npm run build` 통과
- 변경 범위가 이슈 범위 안에 있어야 함
- 요청 범위 밖 파일이 수정되지 않아야 함
- `.env` 등 민감한 파일이 포함되지 않아야 함
- 의도하지 않은 `package-lock.json` 변경이 없어야 함
- TypeScript `any`를 사용하지 않아야 함

---

## lint 실행 불가 예외

프로젝트 초기 단계에서 `npm run lint` 스크립트가 아직 설정되지 않은 경우가 있을 수 있습니다.

이 경우 AI 에이전트는 commit/push를 즉시 중단하지 않고 다음 기준을 따릅니다.

- `npm run lint` 실행이 불가능하다는 사실을 보고합니다.
- `npm run build`가 통과하는지 확인합니다.
- 해당 작업이 ESLint 설정 이전의 작업인지 확인합니다.
- 사용자가 명시적으로 `이슈커밋푸시 진행해`라고 지시한 경우, lint 실행 불가를 보고한 뒤 build 통과 상태에서 commit/push를 진행할 수 있습니다.
- 단, ESLint 설정 이슈가 완료된 이후에는 코드 변경 시 `npm run lint` 통과를 필수로 합니다.

### 주의

ESLint 설정과 lint 스크립트가 추가된 이후에는 이 예외를 남용하지 않습니다.

## 푸시 기준

main 브랜치에 push하기 전 다음 조건을 모두 충족해야 합니다.

- 작업이 하나의 이슈 범위 안에 있어야 합니다.
- 커밋 메시지에 실제 GitHub Issue 번호가 있어야 합니다.
- 코드 변경이면 `npm run lint`와 `npm run build`가 통과해야 합니다.
- 문서 작업이면 변경 파일이 문서 파일만이어야 합니다.
- 실패한 빌드는 main에 push하지 않습니다.
- `.env` 등 민감한 파일이 포함되지 않아야 합니다.
- force push는 절대 하지 않습니다.

- 배포 직전 위험한 수정

## 기본 작업 순서

일반적인 작업은 다음 순서로 진행합니다.

1. GitHub Issue 생성 또는 기존 Issue 확인
2. 작업 범위 확인
3. 수정 금지 파일 확인

5. 코드 변경이면 `npm run lint`

10. `git add`
11. `git commit -m "작업 내용 #이슈번호"`
12. `git push`
13. 이슈에 작업 결과 코멘트 작성
14. 이슈 닫기

## AI 에이전트 commit/push 권한 제한

AI 에이전트는 기본적으로 commit/push를 실행하지 않습니다.

AI 에이전트는 사용자의 명시적 지시 없이 다음을 실행하지 않습니다.

- `git commit`
- `git push`
- `git push --force`
- 배포 설정 변경
- 환경변수 변경
- 외부 서비스 연결
- Supabase 프로젝트 생성
- 도메인 설정 변경
- 결제 서비스 연결
- 지도 API 키 설정
- 관리자 계정 생성

기본적으로 AI는 다음까지만 수행합니다.

- 파일 읽기
- 코드 수정
- 문서 수정
- lint/build 실행
- 변경 사항 요약
- 제안 커밋 메시지 작성

---

## 이슈커밋푸시 자동 진행 규칙

기본적으로 AI 에이전트는 사용자의 명시적 지시 없이 commit/push를 실행하지 않습니다.

단, 사용자가 다음과 같이 명시하면 AI 에이전트는 이슈 생성, 검증, 커밋, 푸시까지 진행할 수 있습니다.

### 명령 예시

```txt
이슈커밋푸시 진행해
이 작업 이슈 만들고 커밋하고 푸시까지 해줘
issue commit push까지 진행해
```

이 명령을 받은 경우 AI 에이전트는 해당 작업 범위 안에서만 commit/push 권한을 가집니다.

다음 작업에서 다시 commit/push가 필요하면 사용자가 다시 명시해야 합니다.

---

## 이슈커밋푸시 자동 진행 순서

## 1. 현재 변경 상태 확인

```bash
git status
git diff
```

확인할 것:

- 변경 파일 목록
- 변경 범위
- 의도하지 않은 파일 변경 여부
- `.env` 등 민감한 파일 포함 여부
- 대규모 자동 포맷팅 여부
- 의도하지 않은 lockfile 변경 여부

---

## 2. 작업 범위 확인

변경 파일이 현재 작업 목적과 맞는지 확인합니다.

요청 범위 밖 파일이 섞여 있으면 중단하고 사용자에게 보고합니다.

작업 범위가 애매하면 commit/push하지 않고 사용자에게 확인합니다.

---

## 3. GitHub Issue 생성 또는 확인

GitHub CLI 사용 가능 시 `gh issue create`로 이슈를 생성하거나, 기존 이슈 번호를 확인합니다.

이슈 제목은 작업 내용을 명확히 설명하는 한국어 문장으로 작성합니다.

이슈 본문에는 다음을 포함합니다.

- 목표
- 작업 범위
- 요구사항
- 제외 범위
- 완료 기준
- 확인 방법

`gh` CLI를 사용할 수 없으면 이슈 생성용 제목과 본문을 제안하고 중단합니다.

이미 이슈가 존재한다면 해당 이슈 번호를 사용합니다.

이슈 번호가 불명확하면 commit/push하지 않습니다.

---

## 4. 검증

### 문서만 수정한 경우

```bash
git status
git diff
```

문서 작업에서는 build를 생략할 수 있습니다.

### 코드 수정이 있는 경우

```bash
npm run lint
npm run build
git status
git diff
```

lint 또는 build 실패 시 commit/push하지 않고 사용자에게 보고합니다.

단, ESLint 설정 이전 단계에서 lint 스크립트가 없는 경우에는 “lint 실행 불가 예외” 기준을 따릅니다.

---

## 5. 커밋

커밋 메시지는 다음 형식으로 작성합니다.

```txt
작업 내용 #이슈번호
```

예시:

```txt
자주 확인하는 군산 정보 칩 UI 보정 #22
```

하나의 커밋에는 하나의 작업만 포함합니다.

이슈 번호가 없거나 불명확하면 커밋하지 않습니다.

---

## 6. 푸시

작은 문서 작업과 작은 UI 작업은 main에 직접 push할 수 있습니다.

다음 작업은 별도 브랜치를 사용합니다.

- Supabase 연결
- 지도 기능
- 사진 업로드
- 관리자 페이지
- 로그인/권한 관련 작업
- 대규모 레이아웃 변경
- package 구조 변경
- 배포 직전 위험한 수정

force push는 절대 하지 않습니다.

```txt
- ...

확인
- ...

- ...

- ...

커밋
- <커밋 해시>
```

## 자동 진행 중단 조건

다음 상황에서는 commit/push를 중단하고 사용자에게 보고합니다.

- 작업 범위 밖 파일이 포함됨
- `.env` 또는 비밀 키 파일이 포함됨
- 의도하지 않은 lockfile 변경 발생
- 대규모 자동 포맷팅 발생
- `npm run lint` 실패
- `npm run build` 실패
- 이슈 번호 생성 실패
- 이슈 번호가 불명확함
- 커밋 메시지에 실제 GitHub Issue 번호를 붙일 수 없음
- 사용자가 요청하지 않은 기능이 섞임
- 여러 API 연결이 한 작업에 섞임
- Supabase, 지도, 사진 업로드, 관리자 작업이 작은 UI 작업에 섞임
- 외부 사진, 리뷰, 지도 이미지 무단 사용이 포함됨
- TypeScript `any`가 포함됨
- 실제 데이터처럼 보이는 가짜 수치, 날짜, 실시간 상태가 포함됨
- 실패한 build 상태
- 사용자가 중단을 요청함

사용자가 명시하지 않으면 AI 에이전트는 commit/push를 실행하지 않습니다.

## 절대 금지

- 사용자의 명시적 지시 없는 commit

- force push
- 이슈 없는 큰 작업
- 이슈 번호가 불명확한 커밋

- 문서 작업과 코드 작업을 한 커밋에 섞기
- UI 작업과 API 연결을 한 커밋에 섞기
- Supabase, 지도, 사진 업로드, 관리자 기능을 선제 구현
- 외부 리뷰, 사진, 본문 무단 복사
- 외부 지도 이미지 무단 사용
- 실제 데이터처럼 보이는 가짜 수치 표시
- 실제 행사처럼 보이는 가짜 날짜 표시
- 실제 지도 기능처럼 보이는 허위 UI 구현
- TypeScript `any` 사용
- 실패한 build 상태로 main push
- `.env` 또는 비밀 키 파일 커밋
- 광고 사이트처럼 보이는 방향으로 임의 변경
- 사용자가 제공한 레퍼런스 방향과 충돌하는 대규모 UI 변경
