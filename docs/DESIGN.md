# 군산.com 디자인 규칙 (DESIGN.md)

## 디자인 정체성

군산.com은 **“군산의 모든 것”** 을 목표로 하는 군산 생활·관광 정보 플랫폼입니다.

군산의 맛집, 카페, 병원/약국, 생활업체, 숙박, 관광/여행, 행사·축제, 주차, 동네 정보를 한곳에서 확인할 수 있는 구조를 지향합니다.

군산.com은 단순한 랜딩페이지가 아닙니다.

군산에 사는 사람과 군산에 방문하려는 사람이 모두 사용할 수 있는 **군산 대표 포털형 홈**을 지향합니다.

군산.com은 Dechive의 검증 철학을 지역 생활 정보에 적용합니다.

다만 사용자가 처음 느끼는 인상은 Dechive처럼 어둡거나 고딕적인 분위기가 아니라, 밝고 명확한 지역 정보 포털이어야 합니다.

군산.com은 다음처럼 보여야 합니다.

- 군산 정보를 찾는 대표 입구
- 생활 정보와 여행 정보를 함께 다루는 지역 포털
- 업체 제공 정보와 방문자 확인 정보를 구분하는 신뢰형 정보 플랫폼
- 시장, 시청, 지역 관계자에게 보여줘도 부끄럽지 않은 구조

**디자인 키워드**

명확함 · 신뢰감 · 빠른 탐색 · 포털형 밀도 · 쉬운 비교 · 정보 우선 · 광고 느낌 최소화

## 레퍼런스 이미지 기준

사용자가 제공한 레퍼런스 이미지는 메인 홈 디자인의 기준입니다.

앞으로 메인 홈 디자인은 레퍼런스 이미지의 구조를 우선합니다.

### 레퍼런스에서 반드시 유지할 방향

- 넓은 2단 포털형 상단바
- 긴 캡슐형 검색창
- 오른쪽 날씨/공지/제보/즐겨찾기/사장님 등록 영역
- 두 번째 줄 카테고리 내비게이션
- full-width에 가까운 파노라마 히어로
- 히어로 위 텍스트와 floating quick card
- 히어로 아래 작은 카테고리 카드 묶음
- 중단 3단 정보 블록
  - 지금 많이 찾는 정보
  - 군산 장소 둘러보기
  - 오늘의 군산
- 하단 확장 블록
  - 행사·축제
  - 지도에서 보기
  - 동네별 보기
- 한 화면에 많은 정보가 보이는 포털형 밀도

### 레퍼런스와 다르게 만들면 안 되는 것

- SaaS 랜딩페이지처럼 넓은 여백만 많은 구조
- 블로그 첫 화면처럼 큰 글과 CTA만 있는 구조
- 앱처럼 보이는 모바일 전용 UI
- 감성 포트폴리오 스타일
- 카드 하나하나가 너무 커서 스크롤이 과하게 길어지는 구조
- “준비 중”, “정리 예정” 문구가 반복되는 미완성 화면

## 디자인 목표

군산.com의 디자인은 다음 질문에 답해야 합니다.

- 여기가 군산 정보를 찾는 대표 사이트라는 것이 바로 보이는가?
- 군산에 사는 사람과 군산에 오는 사람이 모두 자신에게 필요한 정보로 빠르게 진입할 수 있는가?
- 검색창이 명확하게 보이는가?
- 맛집, 카페, 병원/약국, 생활업체, 숙박, 여행, 행사 정보로 바로 들어갈 수 있는가?
- 장소 정보와 검증 상태가 한눈에 보이는가?
- 업체 제공 정보와 방문자 확인 정보가 구분되는가?
- 광고 사이트처럼 보이지 않는가?
- 데스크탑에서 여러 정보를 한눈에 비교할 수 있는가?
- 모바일과 태블릿에서도 화면이 지나치게 길어지지 않는가?
- 시장, 시청, 지역 관계자에게 보여줘도 신뢰가 깨지지 않는가?

## 핵심 브랜드 문장

군산.com의 가장 큰 브랜드 방향은 다음 문장입니다.

> 군산.com = 군산의 모든 것

### 보조 브랜드 문장

- 군산에 산다면, 군산.com.
- 군산에 여행 가고 싶다면, 군산.com.
- 사는 사람도, 오는 사람도 군산.com에서 시작합니다.
- 군산에서 어디 갈지 고민될 때, 군산.com.
- 군산의 맛집, 생활업체, 여행, 행사 정보를 한곳에서 확인하세요.
- 업체 제공 정보와 방문자 확인 정보를 나누어 보여드립니다.
- 정보가 달라졌다면 제보해주세요.

### 메인 카피

| 구역             | 문구                                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------------------------- |
| 큰 문구          | 군산의 모든 것, 군산.com                                                                                 |
| 보조 문구        | 밥 먹을 곳, 쉬어갈 카페, 급하게 찾는 병원, 주차 되는 곳까지. 군산에서 필요한 정보를 한곳에서 확인하세요. |
| 검증 보조 문구   | 사장님이 알려준 정보와 실제 다녀온 사람이 확인한 정보를 나눠서 보여줍니다.                               |
| 검색 placeholder | 맛집, 병원, 카페, 행사 등 군산의 정보를 검색해 보세요                                                    |

### 문구 원칙

- 쉽고 짧게 쓴다.
- 행정 문서처럼 딱딱하게 쓰지 않는다.
- 광고 문구처럼 과장하지 않는다.
- 실제 기능이 없는데 가능한 것처럼 쓰지 않는다.
- 메인 화면에서 “준비 중”, “정리 예정”, “연결 예정” 문구를 반복하지 않는다.
- 실제 데이터가 없는 경우에도 허위 수치, 허위 날짜, 허위 실시간 표현을 쓰지 않는다.

## 메인 홈과 장소 탐색 페이지 역할 분리

## 메인 홈

메인 홈은 군산 대표 포털형 입구입니다.

역할:

- 군산에 사는 사람과 군산에 오는 사람이 모두 처음 들어오는 화면
- 군산의 주요 정보 카테고리로 빠르게 이동하는 화면
- 오늘의 군산, 행사·축제, 지도 보기, 동네별 보기 같은 확장 가능성을 보여주는 화면
- 군산.com이 “군산의 모든 것”을 담는 사이트라는 인상을 주는 화면
- 너무 길게 설명하지 않고, 한 화면에 많은 정보를 압축해서 보여주는 화면

메인 홈에 포함할 수 있는 영역:

- Header
- 대표 히어로
- 통합 검색
- 주요 카테고리 진입
- 지금 많이 찾는 정보
- 군산 장소 둘러보기
- 오늘의 군산
- 행사·축제
- 지도에서 보기
- 동네별 보기
- 정보 제보하기

## /places

`/places`는 필터 중심 장소 탐색 페이지입니다.

역할:

- 장소 목록 탐색
- 카테고리/동네/상태 필터
- 장소 카드 비교
- 검증 상태 확인
- 장소 상세 페이지 진입

`/places`에서는 데스크탑 3단 레이아웃을 사용할 수 있습니다.

```txt
[좌: 필터 패널] | [중: 장소 목록] | [우: 사이드바 / 미리보기]
```

## /places/[slug]

장소 상세 페이지는 장소 하나에 대한 신뢰 정보의 중심입니다.

역할:

- 장소명, 카테고리, 지역, 주소 등 기본 정보 표시
- 업체 제공 정보 표시
- 방문자 확인 정보 표시
- 확인 필요 정보 표시
- 정보 차이 제보 CTA 제공

홈의 장소 카드는 모달보다 상세페이지 이동을 우선합니다.

모달은 나중에 지도 핀 클릭이나 빠른 미리보기 용도로만 검토합니다.

## 레이아웃 원칙

군산.com은 데스크탑 정보 탐색을 기본으로 합니다.

모바일에서도 잘 보여야 하지만, 모바일 앱처럼 보이는 구조를 우선하지 않습니다.

기본은 웹사이트이며, 모바일과 태블릿은 반응형으로 자연스럽게 대응합니다.

### 기본 원칙

- 메인 홈은 포털형 섹션 레이아웃을 기본으로 한다.
- 레퍼런스 이미지처럼 상단, 히어로, 카테고리, 중단 정보 블록이 한 화면 안에서 밀도 있게 보여야 한다.
- 데스크탑에서는 넓은 화면을 활용해 여러 섹션을 한눈에 보여준다.
- 태블릿에서는 2열 또는 compact grid를 적극 사용한다.
- 모바일에서는 자연스럽게 쌓되, 카드가 너무 커져 스크롤이 과도하게 길어지지 않게 한다.
- 앱처럼 보이는 하단 탭바를 사용하지 않는다.
- 정보 비교가 쉬운 카드형 레이아웃을 사용한다.
- 설명 문장을 길게 넣어 카드 높이를 늘리지 않는다.

### 메인 홈 권장 순서

1. 2단 Header
2. full-width 히어로
3. 카테고리 카드
4. 중단 3단 정보 영역
   - 지금 많이 찾는 정보
   - 군산 장소 둘러보기
   - 오늘의 군산

5. 하단 확장 영역
   - 행사·축제
   - 지도에서 보기
   - 동네별 보기

6. 제보/사장님 등록 CTA

## Header 디자인 규칙

Header는 군산.com의 첫 진입점입니다.

레퍼런스 이미지처럼 2단 포털형 구조를 기준으로 합니다.

### 1단 Header

포함 요소:

- `군산.com` 로고
- 긴 캡슐형 검색창
- 원형 검색 버튼
- 군산 날씨
- 공지사항
- 제보하기
- 즐겨찾기
- 사장님 등록

디자인 원칙:

- 전체 배경은 흰색 또는 아주 연한 톤을 사용한다.
- 로고는 왼쪽에 명확하게 배치한다.
- 검색창은 중앙에서 가장 눈에 잘 보여야 한다.
- 검색창은 긴 캡슐형으로 구성한다.
- 검색 버튼은 오른쪽에 원형 또는 작은 강조 버튼으로 둔다.
- 오른쪽 utility 메뉴는 텍스트 링크처럼 단정하게 둔다.
- 사장님 등록만 버튼 형태로 약하게 강조한다.
- Header 안에 “준비 중” 문구를 노출하지 않는다.
- 버튼과 링크가 너무 크면 안 된다.

### 2단 Navigation

포함 가능한 항목:

- 홈
- 정보
- 맛집
- 카페
- 병원/약국
- 생활업체
- 숙박
- 여행
- 행사·축제
- 커뮤니티

디자인 원칙:

- 한 줄 포털 탭처럼 보여야 한다.
- 모든 메뉴를 큰 버튼처럼 만들지 않는다.
- 텍스트는 짧고 명확하게 유지한다.
- 현재 연결되지 않은 항목도 “준비 중” 라벨을 반복하지 않는다.
- 실제 연결이 없으면 안전한 링크, 비활성 스타일, 또는 추후 연결 가능한 구조로 둔다.
- 모바일에서는 자연스럽게 줄바꿈하거나 가로 스크롤을 허용할 수 있다.

## 히어로 디자인 규칙

히어로는 군산 대표 포털이라는 인상을 만드는 영역입니다.

레퍼런스처럼 full-width 파노라마 배너에 가깝게 구성합니다.

### 방향

- 히어로 이미지는 박스 안 카드처럼 보이면 안 된다.
- 이미지 배경은 화면 가로폭을 넓게 사용한다.
- 텍스트와 floating card는 내부 max-width 컨테이너 안에 둔다.
- 히어로 문구는 이미지보다 먼저 읽혀야 한다.
- 관광 포스터처럼 과장하면 안 된다.
- SaaS 랜딩페이지처럼 큰 CTA 중심으로 만들지 않는다.

### 포함 요소

- 큰 문구
- 짧은 설명
- 검색 또는 검색 보조 UI
- 빠른 정보 floating card
- 군산을 암시하는 파노라마 비주얼

### 비주얼 원칙

- 군산의 지역성이 느껴져야 한다.
- 특정 실제 건물, 실제 상호를 허위로 생성하지 않는다.
- AI 생성 이미지는 실제 사진처럼 오해되지 않게 사용한다.
- 검색창과 문구를 가리면 안 된다.
- 이미지가 없는 상태에서도 CSS fallback으로 구조가 유지되어야 한다.

## 카테고리 카드 디자인 규칙

카테고리는 아이콘 메뉴가 아니라 레퍼런스처럼 작은 카드형 진입 영역이어야 합니다.

### 표시 카테고리

- 맛집
- 카페
- 병원/약국
- 생활업체
- 숙박
- 관광/여행
- 행사·축제
- 네일/뷰티

### 디자인 원칙

- 카드 높이는 모든 카테고리에서 동일하게 유지한다.
- 카드 안 부제목/설명은 넣지 않는다.
- 카테고리명은 카드 왼쪽 중앙에 배치한다.
- 카테고리명은 왼쪽 정렬한다.
- 이미지는 카드 오른쪽에 작게 배치한다.
- 이미지가 카드 높이를 키우면 안 된다.
- 카드가 너무 커지면 안 된다.
- 카드 간격은 촘촘하게 유지한다.
- 이미지와 텍스트의 균형을 맞춘다.
- “준비 중” 라벨을 카드 안에 반복하지 않는다.
- 실제 기능이 없는 항목은 과장하지 않고 안전하게 처리한다.

### 이미지 원칙

- 카테고리 이미지는 작은 보조 비주얼이다.
- 이미지가 카테고리명보다 더 튀면 안 된다.
- 직접 제작하거나 사용 권한이 확인된 이미지만 사용한다.
- WebP 사용을 우선한다.
- alt 문구를 작성한다.

## 지금 많이 찾는 정보 디자인 규칙

“지금 많이 찾는 정보” 또는 “자주 확인하는 군산 정보” 영역은 긴 설명 카드가 아니라 compact chip/grid 구조를 사용합니다.

### 역할

- 군산 주민과 여행자가 자주 확인하는 정보를 빠르게 보여준다.
- 검색이나 상세 기능으로 확장될 수 있는 진입점이다.
- 메인 홈의 세로 길이를 줄이고, 정보 밀도를 높인다.

### 표시 예시

- 군산사랑상품권
- 여객선터미널
- 은파호수공원
- 군산시간여행축제
- 새만금방조제
- 초원사진관
- 군산근대역사박물관
- 군산시청 민원
- 주차장 정보
- 공영주차장

### 디자인 원칙

- 작은 pill/chip 형태로 표시한다.
- 긴 설명 문구를 넣지 않는다.
- chip 안에는 정보명만 표시한다.
- 2열 compact grid를 기본으로 한다.
- 데스크탑, 태블릿, 모바일 모두에서 세로로 너무 길어지지 않게 한다.
- chip 높이는 낮고 일정하게 유지한다.
- 실제 검색량처럼 보이는 표현을 쓰지 않는다.
- “실시간 인기”, “조회수”, “오늘 많이 본” 같은 문구를 쓰지 않는다.
- “준비 중”, “정리 예정”, “연결 예정” 문구를 표시하지 않는다.
- 나중에 선택/검색/필터 연결이 가능하도록 key/label 구조를 고려한다.

## 군산 장소 둘러보기 디자인 규칙

“군산 장소 둘러보기”는 홈의 장소 preview 영역입니다.

### 역할

- 전체 장소 목록이 아니라 대표 장소 일부를 보여준다.
- 사용자가 `/places` 또는 장소 상세페이지로 이동하게 만든다.
- 군산.com이 실제 장소 정보를 정리하는 서비스라는 인상을 준다.

### 표시 개수

- 데스크탑: 4~5개 정도
- 태블릿: 3~4개 정도
- 모바일: 3개 정도 우선

### 카드 정보 우선순위

1. 장소명
2. 카테고리
3. area
4. 상태 배지
5. 짧은 설명
6. 상세 보기

### 디자인 원칙

- 카드 하나가 너무 커지면 안 된다.
- 이미지가 들어갈 수 있지만, 검증 상태보다 더 강조되면 안 된다.
- 장소 카드는 상세페이지로 이동하는 구조를 우선한다.
- 모달은 기본 동작으로 사용하지 않는다.
- 더 보기 버튼은 `/places`로 연결한다.
- 카드 전체가 클릭 가능한 링크여도 된다.
- 광고처럼 보이는 강조 효과를 쓰지 않는다.
- 별점 5개 중심 UI를 쓰지 않는다.

## 오늘의 군산 디자인 규칙

“오늘의 군산”은 날씨, 미세먼지, 일몰, 오늘의 행사 등으로 확장 가능한 정보 영역입니다.

### 현재 우선순위

- 군산 현재 날씨
- 기온
- 날씨 상태
- 나중에 미세먼지, 일몰, 행사 정보로 확장

### 디자인 원칙

- 정보 카드처럼 작고 단정하게 표시한다.
- 실제 API 값이 있으면 표시한다.
- API 실패 시 화면이 깨지지 않게 fallback을 둔다.
- 실제 값이 없으면 가짜 수치를 만들지 않는다.
- “준비 중” 문구를 메인 화면에 반복하지 않는다.
- 날씨 API는 독립된 작업으로 연결할 수 있다.
- 미세먼지, 일몰, 행사 API는 별도 작업으로 분리한다.

## 행사·축제 디자인 규칙

행사·축제 영역은 군산 여행자와 주민 모두에게 중요한 정보 진입점입니다.

### 역할

- 군산에서 열리는 행사, 축제, 공연, 전시, 체험 정보로 확장되는 영역
- 공식 출처 기반 정보 정리 방향을 보여주는 영역

### 디자인 원칙

- 실제 날짜, 장소, 개최 상태를 확인 없이 쓰지 않는다.
- 외부 행사 포스터나 사진을 무단 사용하지 않는다.
- 실제 개최 중인 행사처럼 보이는 허위 표현을 쓰지 않는다.
- “준비 중” 문구를 반복하지 않는다.
- 공식 출처 확인 전에는 확정적인 날짜/상태를 만들지 않는다.
- 행사 API 또는 공공데이터 연결은 별도 작업으로 분리한다.
- 카드 높이를 낮게 유지한다.
- 메인 홈 하단에서 과하게 큰 영역을 차지하지 않는다.

## 지도에서 보기 디자인 규칙

지도는 군산.com의 중요한 확장 기능입니다.

다만 실제 지도 API 연결 전에는 실제 지도 기능처럼 보이게 만들지 않습니다.

### 현재 단계

- CSS 기반 preview 가능
- 선, 점, 핀 형태의 추상적 지도형 비주얼 가능
- 실제 지도 이미지 사용 금지
- 실제 좌표 마커 구현 금지
- 확대/축소 버튼, 현재 위치 버튼 같은 실제 지도 UI 금지

### 디자인 원칙

- “지도 기능이 이미 동작한다”는 오해를 주지 않는다.
- 외부 지도 이미지를 사용하지 않는다.
- data/places.ts의 좌표 값을 실제 지도처럼 시각화하지 않는다.
- 지도 API 연결은 별도 작업으로 분리한다.
- 레퍼런스처럼 하단 카드 중 하나로 compact하게 보여준다.

## 색상 시스템

군산.com은 밝고 명확해야 합니다.

| 용도                  | 방향                                                |
| --------------------- | --------------------------------------------------- |
| 전체 배경             | 흰색 또는 아주 연한 회색 (`slate-50`)               |
| 카드 배경             | 흰색                                                |
| 본문 텍스트           | 진한 slate 계열 (`slate-800` ~ `slate-900`)         |
| 보조 텍스트           | `slate-500` ~ `slate-600`                           |
| 경계선                | `slate-200` 수준                                    |
| 포인트 컬러           | 군산 바다 느낌의 파랑/청록 (`sky-500` ~ `teal-500`) |
| 검색 버튼             | teal 또는 sky 계열                                  |
| 확인 상태 배지        | 초록 (`green-500` ~ `emerald-500`)                  |
| 미확인 / 확인 필요    | 회색 또는 노랑 (`slate-400`, `yellow-400`)          |
| 현장 차이 제보 있음   | 주황 (`orange-500`)                                 |
| 폐업/이전 확인 필요   | 빨강 (`red-500`)                                    |
| 업체 제공 정보 영역   | 파랑 계열 구분선 또는 배경 틴트                     |
| 방문자 확인 정보 영역 | 초록 계열 구분선 또는 배경 틴트                     |

### 색상 원칙

- 너무 화려한 그라디언트를 쓰지 않는다.
- 군산 바다 느낌의 청록/하늘색을 포인트로 사용한다.
- 광고 사이트처럼 원색 버튼을 남발하지 않는다.
- 색상만으로 의미를 전달하지 않는다.
- 텍스트와 배지 레이블을 함께 사용한다.

## 검증 상태 배지 디자인 규칙

`VerificationStatus` 값에 따라 색상과 레이블을 다르게 표시합니다.

| 상태                | 배지 색      | 의미                                |
| ------------------- | ------------ | ----------------------------------- |
| 최근 확인됨         | 초록         | 방문자 또는 관리자가 최근 직접 확인 |
| 방문자 후기         | 초록 연한 톤 | 방문자 후기 존재                    |
| 방문자 사진         | 초록 연한 톤 | 방문자 사진 존재                    |
| 업체 제공           | 파랑         | 사장님 제공 정보                    |
| 정보 수집됨         | 회색         | 기본 정보만 있음                    |
| 확인 필요           | 노랑         | 정보 최신성 불확실                  |
| 현장 차이 제보 있음 | 주황         | 온라인 정보와 실제 차이 제보 있음   |
| 폐업/이전 확인 필요 | 빨강         | 폐업 또는 이전 가능성               |

### 배지 원칙

- 너무 화려하게 만들지 않는다.
- 상태가 한눈에 보이게 한다.
- 색상만으로 의미를 전달하지 않고 텍스트를 함께 표시한다.
- “최근 확인됨”과 “업체 제공”을 절대 같은 의미처럼 보이게 하지 않는다.
- 확인되지 않은 정보를 긍정적으로 보이게 꾸미지 않는다.

## 비주얼 asset 기준

군산.com의 비주얼은 예쁜 배경 장식이 아니라, 사용자가 “군산 정보를 찾는 대표 사이트”라고 느끼게 만드는 역할을 합니다.

### 허용

- 사용자가 직접 제작한 이미지
- AI로 직접 생성한 이미지
- 직접 촬영한 이미지
- 업체/방문자/관리자가 제공한 이미지
- 사용 권한이 확인된 이미지
- 출처와 사용 조건이 확인된 공공기관 이미지
- WebP 등 최적화된 이미지 asset

### 금지

- 외부 사진 무단 다운로드
- 외부 리뷰 이미지 무단 사용
- 외부 지도 이미지 무단 사용
- 업체 사진과 방문자 사진을 섞는 것
- 실제 군산 장소처럼 보이는 허위 AI 이미지
- 실제 상호, 실제 간판, 실제 건물을 허위로 생성하는 것
- 저작권이나 사용 권한이 불확실한 이미지 사용

### 적용 원칙

- 히어로 이미지는 full-width 배너로 사용할 수 있다.
- 카테고리 이미지는 작은 보조 비주얼로 사용한다.
- 행사·축제 이미지는 실제 행사처럼 오해되지 않게 사용한다.
- 지도 preview는 실제 지도 이미지가 아니라 CSS 기반 비주얼을 우선한다.
- 사진이 정보보다 더 중요해 보이면 안 된다.
- 이미지 파일 추가 시 위치, 용량, alt 문구를 확인한다.
- 이미지 추가는 가능한 한 별도 작업으로 분리한다.

4. 핵심 체크 정보

6. 방문자 확인 정보

- 광고 문구 중심 구성
- 출처가 섞인 사진 갤러리
- 확인되지 않은 정보를 확정처럼 표현
- 업체 제공 정보와 방문자 확인 정보 혼합
- 별점 중심 평가 UI

## 정보 분리 원칙

업체 제공 정보와 방문자 확인 정보는 UI에서 반드시 시각적으로 구분합니다.

### 업체 제공 정보

표시 예시:

- 섹션 제목: `업체 제공 정보`
- 색상 방향: 파랑 계열 구분선 또는 배경 틴트
- 의미: 사장님 또는 업체가 제공한 정보

### 방문자 확인 정보

표시 예시:

- 섹션 제목: `방문자 확인 정보`
- 색상 방향: 초록 계열 구분선 또는 배경 틴트
- 의미: 실제 방문자 제보 또는 확인 정보

### 확인 필요 정보

표시 예시:

- 레이블: `확인 필요`
- 색상 방향: 노랑 또는 회색
- 의미: 최신성 또는 정확성이 아직 불확실한 정보

### 금지

- 하나의 카드나 섹션 안에 출처가 다른 정보를 섞지 않는다.
- 업체 제공 사진을 방문자 사진처럼 보이게 하지 않는다.
- 방문자 확인이 없는 정보를 최근 확인된 것처럼 표시하지 않는다.
- 광고/후원 정보를 검증 정보처럼 보이게 하지 않는다.

## 광고·후원 UI 원칙

군산.com은 광고판처럼 보이면 안 됩니다.

### 허용

- 후원 안내
- 운영 응원 문구
- 사장님 등록 안내
- 업체 제공 정보 표시

### 금지

- 광고비를 낸 장소를 검증된 장소처럼 표시
- 광고비를 낸 장소를 무조건 상단 추천
- 검증 배지와 광고 배지 혼동
- 배너 광고 중심 레이아웃
- 과도한 CTA 반복
- 업체 홍보판처럼 보이는 구조

### 사장님 등록 표현

사장님 등록은 광고 등록이 아닙니다.

다음 관점으로 표현합니다.

- 손님이 궁금해하는 정보를 정확히 정리하는 과정
- 업체 제공 정보와 방문자 확인 정보를 구분하는 과정
- 군산.com의 정보 정확도를 높이는 과정

## 모바일 디자인 원칙

모바일에서는 데스크탑 포털 구조를 억지로 유지하지 않습니다.

하지만 모바일에서도 너무 긴 랜딩페이지처럼 늘어지면 안 됩니다.

### 원칙

- Header는 간결하게 줄이되 검색 진입을 유지한다.
- 카테고리 카드는 2열 또는 가로 스크롤을 사용할 수 있다.
- 지금 많이 찾는 정보는 2열 chip grid를 우선한다.
- 장소 preview는 3개 정도만 먼저 보여준다.
- CTA 버튼을 반복하지 않는다.
- 모달이 화면을 과하게 가리지 않게 한다.
- 이미지보다 정보명이 먼저 읽혀야 한다.

### 금지

- 앱처럼 보이는 하단 탭바
- 지나치게 큰 CTA 버튼 반복
- 화면을 가리는 과한 모달
- 검색보다 이미지가 먼저 보이는 구조
- 정보보다 감성 문구가 더 커지는 구조
- 카드 하나하나가 너무 커서 스크롤이 길어지는 구조

## 최종 디자인 기준

군산.com은 예쁜 사이트보다 **쓸모 있는 사이트**여야 합니다.

사용자가 들어왔을 때 가장 먼저 느껴야 하는 것은 다음입니다.

- 군산 정보를 찾는 곳이구나
- 군산에 사는 사람도, 군산에 여행 오는 사람도 쓸 수 있구나
- 검색하기 쉽구나
- 장소를 비교하기 쉽구나
- 정보 상태가 보이는구나
- 업체 말과 방문자 확인이 구분되는구나
- 내가 아는 정보가 틀리면 제보할 수 있구나
- 군산의 생활 정보와 여행 정보가 한곳으로 모일 수 있겠구나

군산.com의 큰 비전은 “군산의 모든 것”입니다.

하지만 디자인은 항상 다음 순서를 지킵니다.

1. 정보 우선
2. 신뢰 우선
3. 빠른 탐색 우선
4. 레퍼런스형 포털 밀도 유지
5. 광고 느낌 최소화
