# 군산.com 구현 기준

## 목적

이 문서는 군산.com을 장소 목록 중심 포털이 아니라 **Course 중심 여행 큐레이션 플랫폼**으로 구현하기 위한 도메인, 타입, 데이터, 페이지 경계를 정의합니다.

현재 문서 작업은 방향 기준만 바꿉니다. Course 타입, seed 데이터, UI 코드는 별도 이슈에서 구현합니다.

## 제품 모델

```txt
Course → CourseStop → Place
```

- `Course`는 사용자가 선택하는 여행 단위입니다.
- `CourseStop`은 이동 순서와 방문 맥락을 설명합니다.
- `Place`는 주소록 항목이 아니라 코스를 구성하고 검증하는 장소 정보입니다.
- 전체 장소 DB는 코스와 추천을 뒷받침하는 기반입니다.

## 구현 우선순위

1. Course 타입 설계
2. CourseStop 타입 설계
3. 초기 코스 seed 데이터
4. 코스 선택 중심 메인 UI
5. 코스 상세 페이지
6. Place에 코스 맥락 추가
7. 전체 장소 검색
8. 생활 정보 확장

타입, seed, UI를 하나의 이슈로 묶지 않습니다.

## 초기 MVP 콘텐츠

### 코스

- 군산 처음 반나절
- 군산 처음 1박 2일
- 군산 처음 2박 3일
- 아이랑 군산
- 비 오는 날 군산
- 선유도

### 압축 선택지

- 군산 필수 5
- 군산 대표 먹거리 5
- 군산다운 카페 5

각 숫자는 실제 데이터 수나 인기 순위가 아니라 편집 범위를 의미합니다. 항목을 억지로 채우지 않습니다.

## Course 개념

문서 기준 필드:

```ts
interface Course {
  title: string;
  duration: string;
  target: string;
  theme: string;
  summary: string;
  stops: CourseStop[];
  estimatedTime: string;
  transportType: string;
  recommendedFor: string[];
  notRecommendedFor: string[];
  routeNote: string;
  alternativeStops: string[];
}
```

이 코드는 설계 예시이며 이번 문서 작업에서 실제 타입 파일을 만들지 않습니다. 구현 시 문자열 union, 식별자, 시간 표현, optional 여부를 별도 이슈에서 검토합니다.

### 필드 의미

- `title`: 사용자가 일정과 목적을 바로 이해하는 이름
- `duration`: 반나절, 1박 2일 같은 여행 단위
- `target`: 처음 방문, 아이 동반 등 핵심 대상
- `theme`: 근대거리, 선유도, 비 오는 날 같은 주제
- `summary`: 이 코스를 선택할 이유
- `stops`: 순서가 있는 정류장
- `estimatedTime`: 검증된 범위의 예상 총 시간
- `transportType`: 도보, 자가용, 대중교통 등
- `recommendedFor`: 잘 맞는 사람
- `notRecommendedFor`: 동선이나 환경상 맞지 않는 사람
- `routeNote`: 이동과 운영 시간 주의사항
- `alternativeStops`: 상황에 따른 대체 정류장

## CourseStop 개념

```ts
interface CourseStop {
  placeId?: string;
  placeSlug?: string;
  stopOrder: number;
  recommendedTime: string;
  stayDuration: string;
  whyHere: string;
  nextMove: string;
  parkingNote: string;
  fallbackPlace?: string;
}
```

구현 시 `placeId`와 `placeSlug` 중 기준 식별자를 하나로 정리합니다. 둘을 동시에 필수로 만들지 않습니다.

### 필드 의미

- `stopOrder`: 실제 코스 순서
- `recommendedTime`: 방문하기 좋은 시간대와 이유
- `stayDuration`: 권장 체류 시간
- `whyHere`: 군산 여행에서 이 장소를 넣은 이유
- `nextMove`: 다음 장소로 이동하는 방법과 맥락
- `parkingNote`: 주차 여부와 대안
- `fallbackPlace`: 휴무, 대기, 날씨 문제 때 대체할 장소

## Place 확장 기준

기존 Place 모델과 `PlaceCategory`는 별도 타입 작업 전까지 유지합니다. 방향 전환을 이유로 기존 필드를 임의 변경하지 않습니다.

향후 추가 검토할 의미:

- `whyVisit`: 왜 가는가
- `courseSlugs`: 어떤 코스에 들어가는가
- `recommendedStay`: 얼마나 머무는가
- `nextPlaceSuggestions`: 다음 장소
- `alternativePlaceSlugs`: 대체 장소
- `parkingSuitability`: 주차 적합성
- `childSuitability`: 아이 동반 적합성
- `rainyDaySuitability`: 비 오는 날 적합성
- `officialUrl`: 공식 링크
- `lastVerifiedAt`: 마지막 확인 날짜
- `statuses`: 검증 상태 목록

실제 필드명과 타입은 별도 이슈에서 확정합니다.

## 기존 필드 규칙

- 동네/지역은 `area`를 사용합니다.
- `neighborhood`를 사용하지 않습니다.
- 검증 상태 목록은 `statuses`를 사용합니다.
- `verificationStatuses`를 사용하지 않습니다.
- TypeScript `any`를 사용하지 않습니다.
- 불명확한 외부 값은 `unknown`에서 narrowing합니다.

기존 `PlaceCategory`는 현재 코드와 호환성을 위해 유지합니다. 코스 중심 전환은 카테고리를 무리하게 늘리는 작업이 아닙니다.

## 페이지 역할

### `/`

메인 홈은 코스 선택과 필수 선택지를 압축하는 화면입니다.

우선 노출:

- 핵심 히어로 카피
- 반나절 / 1박 2일 / 2박 3일 코스
- 아이랑 / 비 오는 날 / 선유도 코스
- 군산 필수 5
- 대표 먹거리 5
- 군산다운 카페 5

전체 장소 목록과 생활 카테고리는 후순위입니다.

### 향후 코스 목록 경로

예시: `/courses`

- 기간, 대상, 테마 기준 코스 탐색
- 선택지는 압축하며 무한 필터를 만들지 않음
- 실제 경로 생성은 별도 이슈

### 향후 코스 상세 경로

예시: `/courses/[slug]`

- 코스 요약
- 순서가 있는 CourseStop
- 추천/비추천 대상
- 이동 및 주차 메모
- 대체 장소
- 공식 출처와 마지막 확인일

### `/places`

- 전체 장소 DB 탐색
- 코스에서 대체 장소를 찾거나 세부 조건을 확인하는 보조 화면
- 메인 첫 경험이 아님

### `/places/[slug]`

- 장소 기본 정보와 검증 상태
- 왜 가는지
- 포함 코스
- 체류 시간
- 다음 장소와 대체 장소
- 업체 제공 정보 / 방문자 확인 정보 분리

### `/submit`

- 코스 동선 오류, 휴무, 주차, 아이 동반, 현장 차이 제보
- 일반 자유게시판으로 확장하지 않음

### `/owners`

- 업체가 코스 판단에 필요한 정확한 정보를 제공하는 안내
- 단순 광고 등록이나 추천 순위 구매로 표현하지 않음

## 메인 컴포넌트 방향

향후 검토 가능한 역할명:

- `CourseHero`
- `CourseQuickSelect`
- `FeaturedCourses`
- `GunsanEssentialFive`
- `SignatureFoodSection`
- `GunsanCafeSection`
- `PurposeCourseSection`
- `CourseStopCard`

현재 컴포넌트 이름을 이번 문서 작업에서 바꾸지 않습니다. 실제 리팩터링은 별도 이슈입니다.

## 코스 데이터 작성 원칙

각 코스는 다음 질문에 답해야 합니다.

- 누구를 위한 코스인가?
- 왜 이 순서인가?
- 실제로 하루 안에 가능한가?
- 장소 간 이동은 무리하지 않은가?
- 휴무, 대기, 날씨 때 대안이 있는가?
- 아이, 고령자, 보행이 어려운 사람에게 적합한가?
- 자가용과 대중교통 조건은 무엇인가?

검증되지 않은 시간, 거리, 비용을 확정값으로 만들지 않습니다.

## 대표 먹거리 데이터 기준

- 맛을 절대평가하지 않습니다.
- 군산 여행의 상징성, 인증 경험, 코스 연결성을 설명합니다.
- 업체 제공 정보와 군산.com 편집 설명을 분리합니다.
- 호불호와 대기, 주차, 영업 시간 같은 현실 조건을 숨기지 않습니다.

## 군산다운 카페 데이터 기준

- 체인점 여부만으로 배제하지 않지만 관광 추천의 중심은 군산에서 갈 이유가 있는 곳입니다.
- 근대거리, 바다, 골목, 오래된 건물, 선유도 이동 동선 등 군산 맥락을 기록합니다.
- 사진 목적, 아이 동반, 비 오는 날, 주차 조건을 구분합니다.

## 행사와 코스 연결

행사는 독립 목록보다 코스 변형 정보로 우선 사용합니다.

예:

- 해당 날짜에 기존 정류장을 행사로 대체할 수 있는가?
- 행사로 인해 주차와 이동 시간이 달라지는가?
- 공식 일정과 장소가 확인되었는가?
- 마지막 확인일은 언제인가?

확인되지 않은 행사 날짜를 seed 데이터에 넣지 않습니다.

## API 경계

API는 코스와 장소를 보강하는 보조 수단입니다.

허용 목적:

- 공식 운영 시간과 휴관일 확인
- 행사 기간과 장소 확인
- 날씨에 따른 코스 주의사항 보강
- 여객선과 이동 정보 확인
- 주소와 공식 링크 검증

금지 목적:

- 장소 수를 빠르게 늘리기 위한 무차별 수집
- 외부 설명문과 이미지를 그대로 복제
- API 결과를 자동 추천 순위로 사용
- 갱신 기준 없이 실시간 상태 표시

하나의 API는 하나의 작업으로 분리하고 `lib/` 또는 `lib/api/`에 둡니다.

## 출처와 검증 모델

다음 의미를 UI와 데이터에서 구분합니다.

- 공식 출처 정보
- 군산.com 편집 추천
- 업체 제공 정보
- 방문자 확인 정보
- 관리자 확인 정보
- 확인 필요
- 현장 차이 제보

추천은 검증 상태가 아니며 공식 출처는 편집 추천을 의미하지 않습니다.

## 이미지 원칙

- 직접 제작, 직접 촬영, 제공받은 이미지, 사용권한이 확인된 이미지만 사용합니다.
- 외부 리뷰와 지도 서비스 이미지를 저장하지 않습니다.
- 실제 군산 장소처럼 오해되는 허위 이미지를 생성하지 않습니다.
- Course 썸네일도 실제 장소와 동선을 오해하게 만들지 않아야 합니다.
- 이미지의 출처와 alt를 관리합니다.

## 사업화 데이터 원칙

코스 안의 유료 입점은 편집 추천과 분리합니다.

- 유료 여부를 명시
- 결제가 검증 상태를 바꾸지 않음
- 결제가 자동으로 핵심 정류장 순서를 결정하지 않음
- 코스 맥락과 실제 방문 조건을 충족하는지 별도 검토

## 모바일 구현 기준

- 첫 화면에 검색, 코스 선택, 필수 5 일부만 압축 노출
- 긴 장소 목록 금지
- 코스 카드는 2열 compact grid 또는 짧은 가로 스크롤
- 불필요한 CTA 반복 금지
- 전체 탐색은 하위 페이지로 이동

## 작업 순서

권장 별도 이슈:

1. 코스 중심 방향 전환 문서 정리
2. Course / CourseStop 타입 설계
3. 초기 코스 seed 데이터 작성
4. 군산 필수 5 콘텐츠 초안
5. 대표 먹거리 5 콘텐츠 초안
6. 군산다운 카페 5 콘텐츠 초안
7. 메인 코스 선택 UI
8. 코스 상세 페이지
9. Place 상세의 코스 맥락 보강
10. 공식 출처 검증 API 단일 연결

## 완료 기준

코스 관련 구현은 다음을 만족해야 합니다.

- Course가 Place 목록보다 먼저 선택됨
- 각 CourseStop에 `whyHere`와 `nextMove`가 있음
- 검증되지 않은 시간과 상태를 만들지 않음
- 대체 장소와 제한 조건이 있음
- 공식 출처와 편집 추천이 구분됨
- 업체 제공과 방문자 확인이 분리됨
- 모바일에서 선택지가 압축됨

## 금지

- Course, seed, UI, API를 한 이슈에 구현
- 카테고리 수를 늘리는 것으로 방향 전환을 대체
- 장소를 많이 보여주는 것을 성공 기준으로 사용
- 맛집 순위와 별점 중심 모델
- 체인 카페를 관광 추천 중심에 무맥락으로 노출
- 외부 데이터 자동 수집으로 추천을 생성
- 가짜 날짜, 시간, 거리, 비용, 실시간 상태 생성
