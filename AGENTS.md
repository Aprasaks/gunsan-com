# 군산.com 작업 규칙

## 프로젝트 개요

군산.com은 **“군산의 모든 것”** 을 목표로 하는 군산 생활·관광 정보 플랫폼입니다.

군산의 맛집, 카페, 병원/약국, 생활업체, 숙박, 관광/여행, 행사·축제, 주차, 동네 정보를 한곳에서 확인할 수 있는 구조를 지향합니다.

군산.com은 단순히 정보를 많이 모으는 사이트가 아닙니다.

군산에서 사람들이 실제로 찾고, 비교하고, 확인하고 싶어 하는 정보를 보기 쉽게 정리하는 지역 정보 포털입니다.

**핵심 브랜드 방향**

> 군산.com = 군산의 모든 것

**핵심 모토**

> 하나라도 물 한 방울이 들어갈 틈이 있다면 헤집고 들어가서 검증한다.

이 프로젝트는 앱도 아니고 모바일 전용 웹도 아닙니다.

기본은 데스크탑 웹사이트이며, 모바일과 태블릿에서도 자연스럽게 반응형으로 보여야 합니다.

군산.com은 군산 시민과 군산에 방문하려는 여행자 모두를 대상으로 합니다.

## 현재 제품 방향

군산.com은 초기 placeholder 사이트에서 벗어나, **레퍼런스 이미지 기준의 군산 대표 포털 홈**으로 전환하고 있습니다.

현재 작업의 핵심은 다음과 같습니다.

- 레퍼런스 이미지처럼 상단바, 히어로, 카테고리, 정보 카드가 촘촘하게 보이는 포털형 홈을 만든다.
- “준비 중”, “정리 예정”, “연결 예정” 문구를 화면에 남발하지 않는다.
- 보이는 메뉴와 카드들은 실제 링크, 실제 섹션 이동, 상세페이지 이동, 또는 안전한 비활성 처리로 다룬다.
- 실제 데이터가 없는 곳에 허위 수치, 허위 날짜, 허위 실시간 정보를 만들지 않는다.
- 날씨 API처럼 작고 독립적인 기능은 별도 작업으로 하나씩 연결할 수 있다.
- 관광 API, 행사 API, 지도 API, DB 저장 등 큰 기능은 한 번에 붙이지 않고 별도 작업으로 분리한다.
- 시장, 시청, 지역 관계자에게 보여줘도 신뢰가 깨지지 않는 정보 구조를 지향한다.

- 광고처럼 보이는 정보와 실제 방문 정보가 구분되는가?
- 군산 여행객이 오늘 무엇을 할 수 있는지 한곳에서 확인할 수 있는가?
- 군산 행사, 축제, 공연, 전시 정보를 한곳에서 확인할 수 있는가?

## 현재 개발 단계

현재 군산.com은 **레퍼런스형 포털 홈 구축 단계**입니다.

초기 v0.1의 더미 데이터 기반 UI와 장소 정보 구조는 유지하되, 메인 홈은 사용자가 제공한 레퍼런스 이미지를 기준으로 더 실제 포털처럼 보이게 고도화합니다.

### 현재 단계에서 구현할 수 있는 것

- 레퍼런스형 2단 상단바
- 군산 대표 full-width 히어로 배너
- 카드형 카테고리 진입 UI
- 자주 확인하는 정보 chip/grid UI
- 군산 장소 둘러보기 preview
- 장소 카드에서 상세페이지로 이동하는 구조
- 오늘의 군산 영역
- 군산 현재 날씨 API 단일 연결
- 행사·지도·동네 영역의 UI 구조
- 직접 제작하거나 사용 권한이 확인된 이미지 asset 적용
- 기존 더미 데이터 기반 장소 목록/상세 페이지
- 업체 제공 정보와 방문자 확인 정보의 UI 분리
- 제보하기 페이지
- 사장님 등록 안내 페이지
- 모바일/태블릿/데스크탑 반응형 조정

### 현재 단계에서 하지 말 것

다음 기능은 사용자가 명시적으로 별도 작업으로 요청하지 않는 한 구현하지 않습니다.

- Supabase 연결
- DB 저장
- 사진 업로드
- 관리자 페이지
- 로그인
- 결제
- 광고 상품
- 사장님 대시보드
- 자유게시판
- 행사/축제 캘린더 실제 구현
- 지도 SDK 연결
- 좌표 기반 실제 지도 마커 구현
- 관광 API 대량 연동
- 행사 API 대량 연동
- 외부 데이터 무단 크롤링
- 외부 리뷰/사진/본문 복사
- 여러 API를 한 번에 연결하는 작업

### API 연결 원칙

API는 한 번에 여러 개 붙이지 않습니다.

허용되는 방식:

1. 하나의 API를 하나의 작업으로 분리한다.
2. API 호출 코드는 컴포넌트 안에 길게 직접 쓰지 않고 `lib/` 또는 `lib/api/` 쪽으로 분리한다.
3. 실패 상태와 로딩 상태를 안전하게 처리한다.
4. 실제 값이 없으면 임의 수치를 만들지 않는다.
5. 화면이 깨지지 않게 fallback을 둔다.

현재 우선 허용 가능한 API:

- 군산 현재 날씨 API

나중에 별도 검토할 API:

- 관광 API
- 행사/축제 API
- 지도 API
- 공공데이터 기반 업소/주차/관광 데이터

## 레퍼런스형 포털 홈 원칙

사용자가 제공한 레퍼런스 이미지는 메인 홈의 주요 시각 기준입니다.

### 상단바

상단바는 2단 포털 구조를 기준으로 합니다.

1단:

- 군산.com 로고
- 긴 캡슐형 검색창
- 날씨
- 공지사항
- 제보하기
- 즐겨찾기
- 사장님 등록

2단:

- 홈
- 정보
- 맛집
- 카페
- 병원/약국
- 생활업체
- 숙박
- 여행
- 행사·축제
- 커뮤니티

원칙:

- 상단바에 “준비 중” 문구를 길게 노출하지 않는다.
- 실제 기능이 없는 항목은 조용한 비활성 처리, 안전한 링크, 또는 이후 연결 가능한 구조로 둔다.
- 사장님 등록만 적당히 강조한다.
- 모든 메뉴를 버튼 박스처럼 만들지 않는다.
- 포털 탭처럼 얇고 단정하게 보이게 한다.

### 히어로

히어로는 박스 안 이미지 카드가 아니라 **full-width 파노라마 배너**처럼 보여야 합니다.

원칙:

- 이미지 자체는 화면 가로폭을 넓게 사용한다.
- 텍스트, 검색, floating card는 내부 max-width 컨테이너 안에 둔다.
- 히어로가 랜딩페이지의 큰 카드처럼 보이면 안 된다.
- 검색과 핵심 문구가 이미지보다 먼저 읽혀야 한다.
- 관광 홍보 포스터처럼 과장되면 안 된다.

### 카테고리

카테고리는 아이콘 메뉴가 아니라 작은 카드형 진입 영역으로 보여야 합니다.

원칙:

- 카테고리 카드 안 부제목은 길게 넣지 않는다.
- 카드 높이를 통일한다.
- 카테고리명은 왼쪽 중앙에 배치한다.
- 이미지는 오른쪽에 작게 배치한다.
- 준비 중 문구를 카드 안에 길게 붙이지 않는다.
- 실제 기능이 없는 항목은 안전한 링크 또는 비활성 처리한다.

표시 카테고리:

- 맛집
- 카페
- 병원/약국
- 생활업체
- 숙박
- 관광/여행
- 행사·축제
- 네일/뷰티

### 자주 확인하는 정보

“자주 확인하는 정보” 또는 “지금 많이 찾는 정보” 영역은 긴 설명 카드가 아니라 compact chip/grid 구조를 우선합니다.

원칙:

- “준비 중”, “정리 예정”, “연결 예정” 문구를 사용하지 않는다.
- 실제 검색량처럼 보이는 표현을 만들지 않는다.
- 조회수, 실시간 인기, 오늘 많이 본 같은 표현을 쓰지 않는다.
- 정보명 중심의 짧은 chip UI를 사용한다.
- 나중에 선택/검색/필터 연결이 쉽도록 key/label 구조로 관리한다.

예시 항목:

- 군산사랑상품권
- 여객선터미널
- 은파호수공원
- 군산시간여행축제
- 새만금방조제
- 초원사진관
- 군산근대역사박물관
- 군산시청 민원
- 주차장 정보
- 공영주차장

### 군산 장소 둘러보기

홈의 장소 카드는 preview 역할입니다.

원칙:

- 홈에서는 일부 장소만 보여준다.
- 카드 클릭 시 모달보다 상세페이지 이동을 우선한다.
- 더 보기 버튼은 `/places`로 연결한다.
- 각 장소는 고유 URL을 가져야 한다.
- 모달은 나중에 지도 핀이나 빠른 미리보기 용도로만 검토한다.
- 장소 상세페이지는 신뢰 정보의 중심이다.

구조:

- 홈: 미리보기
- `/places`: 탐색
- `/places/[slug]`: 상세 정보
- `/submit`: 제보
- `/owners`: 사장님 정보 등록 안내

### 오늘의 군산

오늘의 군산은 날씨, 미세먼지, 일몰, 행사 등으로 확장할 수 있습니다.

현재 우선순위:

- 상단바 또는 오늘의 군산 영역에 군산 현재 날씨 API를 단일 작업으로 연결할 수 있다.
- 실제 값이 없으면 가짜 수치를 만들지 않는다.
- API 실패 시 화면이 깨지지 않게 처리한다.

### 행사·축제

행사·축제 정보는 공식 출처 기반으로 정리하는 방향을 유지합니다.

원칙:

- 실제 날짜, 개최 상태를 확인 없이 쓰지 않는다.
- 외부 행사 포스터나 사진을 무단 사용하지 않는다.
- 실제 개최 중인 행사처럼 보이는 허위 표현을 쓰지 않는다.
- 관광 API나 행사 API 연결은 별도 작업으로 분리한다.

### 지도

지도는 중요한 확장 기능이지만, 실제 지도 API 연결은 별도 작업으로 분리합니다.

원칙:

- 실제 지도 API를 붙이지 않았는데 실제 지도 기능처럼 보이게 만들지 않는다.
- 확대/축소 버튼, 현재 위치 버튼, 실제 좌표 마커처럼 보이는 UI는 지도 API 작업 전에는 만들지 않는다.
- CSS 기반 preview는 가능하다.
- 지도 API 연결은 별도 작업으로 진행한다.

## 광고·후원 원칙

군산.com은 광고판이 아닙니다.

후원이나 기부는 가능하지만, 신뢰 구조를 훼손하면 안 됩니다.

- 후원/기부는 “군산.com 운영을 응원하는 방식”으로만 다룹니다.
- 광고비를 냈다고 검증 상태가 좋아지면 안 됩니다.
- 광고비를 냈다고 추천 순위가 올라가면 안 됩니다.
- 광고/후원/업체 제공 정보는 반드시 표시합니다.
- 업체 등록은 “광고 등록”이 아니라 “손님이 궁금해하는 정보를 정확히 정리하는 과정”으로 다룹니다.
- 시장, 시청, 지역 파트너에게 보여줄 수 있는 신뢰 구조를 유지합니다.

## 공통 코드 규칙

- TypeScript `any` 절대 금지
- 타입이 불명확하면 `unknown`을 사용하고 narrowing 한다.
- 타입은 가능한 한 `types/` 폴더에 분리한다.
- 컴포넌트 내부에 임시 타입을 남발하지 않는다.
- 데이터 모델은 도메인별 타입 파일에서 관리한다.
- 업체 제공 정보와 방문자 확인 정보는 절대 섞지 않는다.
- 확인되지 않은 정보는 긍정적으로 꾸미지 않는다.
- 외부 사이트의 사진, 리뷰, 본문은 무단 복사하지 않는다.
- 한 커밋에는 하나의 작업만 포함한다.
- 작업 범위 밖의 파일은 수정하지 않는다.
- 사용자가 요청하지 않은 기능을 미리 구현하지 않는다.
- 큰 기능은 한 번에 구현하지 않고 작은 작업으로 나눈다.
- API는 하나씩 연결한다.
- 기능 연결 전에는 UI와 데이터 경계를 먼저 정리한다.
- API 호출 코드는 컴포넌트 안에 길게 직접 쓰지 않고 `lib/` 또는 `lib/api/` 쪽으로 분리한다.
- 코드 변경 후 `npm run lint`와 `npm run build`를 확인한다.

## 이슈·커밋·푸시 작업 기준

이슈 생성, 커밋 메시지, 푸시 전 확인, 브랜치 전략, 작업 순서, AI 권한 기준은 `docs/WORKFLOW.md`를 따른다.

요약:

- 하나의 이슈는 하나의 작업 단위
- 하나의 커밋은 하나의 의도만
- 코드 변경 시 커밋 전 `npm run lint` + `npm run build` 통과 필수
- 문서만 수정한 경우 `npm run build`는 생략 가능
- 이슈 없는 큰 작업 금지
- AI는 사용자 명시적 지시 없이 commit/push를 실행하지 않는다.
- 사용자가 `이슈커밋푸시 진행해`라고 명시하면 이슈 생성 → 검증 → 커밋 → 푸시 순서로 진행할 수 있다.

### 대화용 작업 번호와 GitHub Issue 번호 구분

ChatGPT 대화에서 부르는 작업 번호는 임시 진행 번호일 수 있다.

실제 커밋 메시지의 `#번호`는 GitHub Issue 번호를 따른다.

GitHub Issue 번호가 불명확하면 커밋 메시지에 임의 번호를 붙이지 않는다.

사용자가 “이슈 번호 없이 작업명 기준으로 진행”한다고 말한 경우, 작업명 기준으로 보고하되 commit 전 실제 GitHub Issue 번호를 확인한다.
