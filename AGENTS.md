# 군산.com 작업 규칙

## 프로젝트 정의

군산.com은 군산의 모든 장소를 나열하는 정보 포털이 아닙니다.

> 군산.com = 군산을 처음 온 사람이 그대로 따라갈 수 있는 여행 코스와 필수 선택지를 정해주는 군산 여행 큐레이션 플랫폼

처음 온 여행자가 “어디 가야 하지?”, “이성당 다음에는 어디로 가지?”, “1박 2일이면 어떻게 돌지?”를 고민하지 않도록 대표 선택지를 압축하고 실제 이동 순서로 정리합니다.

핵심 문장:

> 군산 처음이라면, 이대로만 다녀도 됩니다.

보조 문장:

> 맛집, 카페, 관광지 고르느라 헤매지 않게 군산에서 가야 할 곳만 코스로 정리했습니다.

“군산의 모든 것”은 장기적인 데이터 축적과 생활 정보 확장 방향으로만 남깁니다. 사용자의 첫 경험과 최상위 브랜드 문장으로 사용하지 않습니다.

## 핵심 제품 원칙

- 군산.com은 선택지를 늘리는 사이트가 아니라 선택을 줄여주는 사이트입니다.
- 모든 장소를 같은 무게로 나열하지 않습니다.
- 장소 수보다 군산에 왔을 때 갈 이유와 코스 안에서의 역할을 우선합니다.
- 검색과 전체 장소 목록보다 코스 선택을 먼저 보여줍니다.
- `Course`가 중심이고 `Place`는 코스를 구성하고 검증하는 재료입니다.
- 대표 선택지는 한 주제당 3~5개 수준으로 압축합니다.
- 맛을 절대평가하지 않고 상징성, 군산다운 경험, 동선, 대체 가능성을 설명합니다.
- 체인 카페보다 군산에서만 갈 이유가 있는 카페를 관광 추천의 중심에 둡니다.
- 생활 정보는 장기 확장 영역이며 초기 메인의 최상단에 두지 않습니다.

## 현재 개발 단계

현재는 **코스 중심 방향 전환 단계**입니다. 기존 장소 데이터와 페이지는 유지하되, 다음 작업부터 코스가 장소 목록보다 먼저 보이도록 작은 이슈로 나누어 전환합니다.

초기 MVP:

- 군산 처음 반나절 코스
- 군산 처음 1박 2일 코스
- 군산 처음 2박 3일 코스
- 아이랑 군산 코스
- 비 오는 날 군산 코스
- 선유도 코스
- 군산 필수 5
- 군산 대표 먹거리 5
- 군산다운 카페 5
- 코스별 장소 상세 구조
- 제보하기
- 사장님 등록 안내

이번 방향 전환만으로 Course 타입, seed 데이터, UI를 선제 구현하지 않습니다. 각각 별도 이슈로 진행합니다.

## 메인 홈 원칙

메인 홈은 장소 DB의 입구가 아니라 여행 결정을 줄이는 화면입니다.

권장 첫 흐름:

1. 투명 헤더와 넓은 군산 히어로
2. “군산 처음이라면, 이대로만 다녀도 됩니다.”
3. 반나절, 1박 2일, 2박 3일 등 코스 선택
4. 군산 필수 5
5. 대표 먹거리 5
6. 군산다운 카페 5
7. 목적별 코스와 전체 검색

검색 placeholder:

> 예: 이성당 다음 코스, 아이랑 군산, 비 오는 날 군산, 선유도 가는 길

상단 메뉴 방향:

- 코스
- 필수장소
- 대표먹거리
- 군산다운카페
- 선유도
- 지도

실제 메뉴 변경은 별도 UI 작업에서 진행합니다.

### 데스크탑

- 비짓제주 레퍼런스처럼 넓은 히어로, 투명 헤더, 검색 오버레이 구조를 사용할 수 있습니다.
- 첫 행동은 카테고리 탐색이나 검색보다 코스 선택이어야 합니다.
- 정보 밀도를 위해 무조건 많은 카드와 카테고리를 노출하지 않습니다.

### 모바일

- 첫 화면은 검색, 코스 선택, 필수 5를 압축한 결정 화면이어야 합니다.
- 전체 맛집·카페·관광지 목록을 메인에 길게 나열하지 않습니다.
- 앱 전용 UI가 아니라 반응형 웹으로 구현하되 긴 스크롤을 피합니다.

## 여행 콘텐츠 기준

### 필수 장소

한일옥, 이성당, 초원사진관, 근대거리, 선유도처럼 군산 여행의 상징이 된 장소는 단순 목록 항목이 아니라 루트의 핵심 경험으로 설명합니다.

각 장소는 다음 질문에 답해야 합니다.

- 왜 가는가?
- 어떤 코스에 들어가는가?
- 얼마나 머무는가?
- 다음 장소는 어디가 좋은가?
- 대체 장소는 무엇인가?
- 주차, 아이 동반, 비 오는 날에 적합한가?

### 대표 먹거리

- “최고의 맛집”, “검증된 맛집” 같은 절대평가를 하지 않습니다.
- 왜 군산 여행에서 들르는지, 어떤 상징과 경험이 있는지 설명합니다.
- 맛의 호불호와 여행 상징성은 분리해 설명합니다.
- 예: “최고의 맛집이라기보다 군산 여행에서 하나의 상징처럼 소비되는 장소입니다.”

### 군산다운 카페

관광 추천은 다음 맥락을 우선합니다.

- 근대거리에서 걷다가 들를 카페
- 선유도/고군산 가는 길에 들를 카페
- 군산 바다가 보이는 카페
- 오래된 건물 분위기가 있는 카페
- 사진 남기기 좋은 카페
- 아이랑 쉬기 좋은 카페
- 비 오는 날 앉아 있기 좋은 카페

체인 카페는 필요할 때 생활 편의 또는 잠깐 쉬기 정보로 분리합니다.

## 검증과 출처 원칙

유지해야 할 정보 구분:

- 공식 출처 기반 정보
- 군산.com 편집 추천
- 업체 제공 정보
- 방문자 확인 정보
- 관리자 확인 정보
- 확인 필요 정보
- 현장 차이 제보 정보

금지:

- 외부 리뷰, 사진, 본문 무단 복사
- 외부 지도/플랫폼 데이터 무단 대량 수집
- 확인되지 않은 정보를 최근 확인된 것처럼 표시
- 가짜 수치, 날짜, 실시간 상태, 조회수, 인기 순위 표시
- 광고/후원 정보를 일반 추천이나 검증 정보처럼 표시
- AI가 만든 허위 장소를 실제 장소처럼 표시

공식 출처 링크와 마지막 확인일을 함께 관리하는 방향을 유지합니다.

## API 원칙

- API는 많은 정보를 자동으로 채우는 수단이 아닙니다.
- 코스와 필수 장소의 운영 시간, 행사 여부, 이동 정보 등을 검증·보강하는 보조 수단입니다.
- 행사 API는 목록을 많이 뿌리는 용도가 아니라 해당 여행 코스에 넣을 수 있는 공식 행사를 확인하는 용도입니다.
- 공식 사이트를 복붙하거나 무단 크롤링하지 않고 군산.com의 코스 맥락으로 재편집합니다.
- 하나의 API는 하나의 작업으로 분리합니다.
- API 코드는 `lib/` 또는 `lib/api/`에 둡니다.
- 로딩, 실패, fallback을 처리하고 실제 값이 없으면 임의 값을 만들지 않습니다.

## 사업화 원칙

군산.com은 가게를 단순 나열하거나 광고비로 추천 순위를 판매하지 않습니다.

가능한 설명 예:

- 군산 처음 코스 추천 입점
- 선유도 가는 길 추천 카페
- 비 오는 날 추천 공간
- 아이랑 코스 추천 식당
- 근대거리 산책 후 들를 카페

입점이나 후원이 코스 적합성, 검증 상태, 추천 순위를 자동으로 높여서는 안 됩니다. 유료 노출은 반드시 표시하고 편집 추천과 분리합니다.

## 기술 및 코드 규칙

- Next.js App Router, TypeScript, Tailwind CSS를 사용합니다.
- TypeScript `any`를 사용하지 않고 불명확한 값은 `unknown`과 narrowing으로 처리합니다.
- 타입은 가능한 한 `types/`에 분리합니다.
- 지역 필드는 `area`, 검증 상태 목록은 `statuses`를 사용합니다.
- `neighborhood`, `verificationStatuses`를 사용하지 않습니다.
- 업체 제공 정보와 방문자 확인 정보를 하나의 타입이나 UI 의미로 섞지 않습니다.
- API 호출을 컴포넌트에 길게 작성하지 않습니다.
- 작업 범위 밖 파일을 수정하지 않습니다.
- 코드 변경 후 `npm run lint`와 `npm run build`를 확인합니다.

기존 `PlaceCategory`는 별도 타입 작업 전까지 유지합니다. Course 방향 전환을 이유로 이번 문서 작업에서 코드 타입을 수정하지 않습니다.

## 이슈·커밋·푸시

- 하나의 이슈는 하나의 작업 단위입니다.
- 문서, UI, 타입, seed 데이터, API 작업을 서로 분리합니다.
- 실제 GitHub Issue 번호가 없거나 불명확하면 commit/push하지 않습니다.
- 사용자의 명시적 지시 없이 commit/push하지 않습니다.
- force push하지 않습니다.
- 자세한 기준은 `docs/WORKFLOW.md`를 따릅니다.

## 절대 하지 말 것

- “군산의 모든 것”을 최상위 사용자 경험으로 되돌리기
- 생활 카테고리와 여행 코스를 같은 우선순위로 메인에 나열하기
- 많은 장소를 보여주는 것을 제품 가치로 삼기

- 맛을 절대평가하거나 광고를 추천처럼 표시하기
- 검증 원칙과 출처 구분을 약화하기
- 사용자 지시 없이 코드, 커밋, 푸시까지 범위를 넓히기
