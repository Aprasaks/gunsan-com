# 군산.com 작업 규칙

## 프로젝트 개요

군산.com은 **“군산의 모든 것”** 을 목표로 하는 지역 정보 웹사이트입니다.

군산의 맛집, 카페, 병원, 숙박, 생활업체, 관광지, 행사 정보를 한곳에서 확인할 수 있는 구조를 지향합니다.

다만 초기 MVP에서 군산의 모든 정보를 한 번에 구현하지 않습니다.
v0.1에서는 장소 정보 UI, 업체 제공 정보와 방문자 확인 정보의 분리, 제보 구조를 위한 기본 화면부터 작게 시작합니다.

군산.com은 단순히 정보를 많이 모으는 사이트가 아니라, 군산에서 사람들이 실제로 찾고 확인하고 싶어 하는 정보를 보기 쉽게 정리하는 사이트입니다.

**핵심 브랜드 방향:**
“군산.com = 군산의 모든 것”

**핵심 모토:**
“하나라도 물 한 방울이 들어갈 틈이 있다면 헤집고 들어가서 검증한다.”

이 프로젝트는 앱도 아니고 모바일 전용 웹도 아닙니다.
기본은 데스크탑 웹사이트이며, 모바일에서도 보기 좋게 반응형으로 구성합니다.

## 군산.com이 검증하려는 틈

군산.com은 검색 결과만으로는 해결되지 않는 지역 정보의 빈틈을 다룹니다.

예시:

- 검색에는 영업 중이라고 뜨는데 실제로 쉬는가?
- 업체 사진과 방문자 사진이 다른가?
- 남자도 네일 케어를 받을 수 있는가?
- 주차가 된다고 하지만 실제로 가능한가?
- 아이와 함께 가도 되는가?
- 사장님이 올린 정보와 방문자 경험이 다른가?
- 광고처럼 보이는 정보와 실제 방문 정보가 구분되는가?
- 군산 여행객이 오늘 무엇을 할 수 있는지 한곳에서 확인할 수 있는가?
- 군산 행사, 축제, 공연, 전시 정보를 한곳에서 확인할 수 있는가?

## 제품 방향

군산.com의 최종 방향은 “군산의 모든 것”입니다.

그러나 개발과 운영은 작게 시작합니다.

초기에는 다음 질문에 답하는 정보부터 시작합니다.

- 군산에서 어디를 가야 하는가?
- 오늘 실제로 여는 곳은 어디인가?
- 주차가 가능한 곳은 어디인가?
- 아이와 함께 가도 되는 곳은 어디인가?
- 남자도 이용 가능한 곳은 어디인가?
- 업체가 제공한 정보와 방문자가 확인한 정보가 어떻게 다른가?

나중에는 다음 영역으로 확장할 수 있습니다.

- 군산 행사
- 군산 축제
- 공연
- 전시
- 체험
- 숙박
- 관광지
- 생활업체
- 구조화된 정보 제보 게시판

단, 확장 기능은 현재 이슈 범위와 v0.1 범위를 넘어서면 구현하지 않습니다.

## 공통 코드 규칙

- TypeScript `any` 절대 금지
- 타입이 불명확하면 `unknown`을 사용하고 narrowing 합니다.
- 타입은 가능한 한 `types/` 폴더에 분리합니다.
- 컴포넌트 내부에 임시 타입을 남발하지 않습니다.
- 데이터 모델은 도메인별 타입 파일에서 관리합니다.
- 업체 제공 정보와 방문자 확인 정보는 절대 섞지 않습니다.
- 확인되지 않은 정보는 “확인 필요”로 표시합니다.
- 외부 사이트의 사진, 리뷰, 본문은 무단 복사하지 않습니다.
- 한 커밋에는 하나의 작업만 포함합니다.
- 작업 범위 밖의 파일은 수정하지 않습니다.
- 사용자가 요청하지 않은 기능을 미리 구현하지 않습니다.
- v0.1 범위 밖의 기능은 문서에만 남기고 코드로 구현하지 않습니다.

## 데이터와 정보 출처 원칙

군산.com은 정보를 많이 모으는 것보다 정보의 상태를 명확히 보여주는 것을 우선합니다.

정보는 다음처럼 구분합니다.

- 업체 제공 정보
- 방문자 확인 정보
- 관리자 확인 정보
- 확인 필요 정보
- 현장 차이 제보 정보

### 금지

- 외부 사이트 리뷰 본문 무단 복사
- 외부 사이트 사진 무단 저장
- 블로그 글 본문 복사
- 지도 서비스 데이터를 무단 대량 수집
- 확인되지 않은 정보를 “최근 확인됨”으로 표시
- 광고/후원 정보를 일반 검증 정보처럼 표시

### 허용 방향

- 직접 작성한 요약
- 공식 출처 링크 표시
- 사장님이 직접 제공한 정보
- 방문자가 직접 제보한 정보
- 사용자가 직접 확인한 정보
- 확인 상태를 명확히 붙인 기본 정보

## 광고·후원 원칙

군산.com은 광고판이 아닙니다.

후원이나 기부는 가능하지만, 신뢰 구조를 훼손하면 안 됩니다.

- 후원/기부는 “군산.com 운영을 응원하는 방식”으로만 다룹니다.
- 광고비를 냈다고 검증 상태가 좋아지면 안 됩니다.
- 광고비를 냈다고 추천 순위가 올라가면 안 됩니다.
- 광고/후원/업체 제공 정보는 반드시 표시합니다.
- 업체 등록은 “광고 등록”이 아니라 “손님이 궁금해하는 정보를 정확히 정리하는 과정”으로 다룹니다.

## 이슈·커밋·푸시 작업 기준

이슈 생성, 커밋 메시지, 푸시 전 확인, 브랜치 전략, 작업 순서, AI 권한 기준은 **[docs/WORKFLOW.md](docs/WORKFLOW.md)** 를 반드시 따릅니다.

요약:

- 하나의 이슈는 하나의 작업 단위
- 하나의 커밋은 하나의 의도만
- 커밋 메시지 형식: `작업 내용 #이슈번호` (한국어)
- #1~#3은 부트스트랩 예외
- #4부터는 GitHub Issue 번호와 반드시 맞춥니다.
- 코드 변경 시 커밋 전 `npm run lint` + `npm run build` 통과 필수
- 문서만 수정한 경우 `npm run build`는 생략 가능
- 이슈 없는 큰 작업 금지
- AI는 사용자 명시적 지시 없이 commit/push를 실행하지 않습니다.
- 사용자가 `이슈커밋푸시 진행해`라고 명시하면 이슈 생성 → 검증 → 커밋 → 푸시까지 진행할 수 있습니다.
- 상세 규칙은 [docs/WORKFLOW.md](docs/WORKFLOW.md)를 따릅니다.

## 작업 범위 원칙

작업 시작 전 다음을 확인합니다.

1. 현재 작업이 어떤 GitHub Issue에 해당하는가?
2. 이번 이슈에서 수정할 파일은 무엇인가?
3. 이번 이슈에서 절대 건드리면 안 되는 파일은 무엇인가?
4. v0.1 범위를 벗어나지 않는가?
5. 사용자가 명시하지 않은 기능을 추가하고 있지 않은가?

작업 범위가 불명확하면 먼저 사용자에게 확인합니다.

## AI 에이전트 권한 기준

AI 에이전트는 기본적으로 다음을 수행합니다.

- 파일 읽기
- 코드 수정
- 문서 수정
- lint/build 실행
- 변경 사항 요약
- 제안 커밋 메시지 작성

AI 에이전트는 사용자의 명시적 지시 없이 다음을 수행하지 않습니다.

- commit
- push
- force push
- 배포 설정 변경
- 환경변수 변경
- 외부 서비스 연결
- Supabase 프로젝트 생성
- 도메인 설정 변경

사용자가 `이슈커밋푸시 진행해`라고 명시한 경우에만 해당 작업 범위 안에서 commit/push까지 진행할 수 있습니다.

## 작업 완료 보고 형식

작업이 끝나면 반드시 아래 형식으로 보고합니다.

```txt
변경 파일
- ...

주요 변경
- ...

확인
- ...

TODO
- ...

제안 커밋 메시지
- 작업 내용 #이슈번호
```

코드 변경이 있는 경우 확인 항목에는 다음을 포함합니다.

```txt
확인
- npm run lint
- npm run build
```

문서만 수정한 경우에는 다음처럼 보고합니다.

```txt
확인
- 문서 작업이므로 npm run build는 생략 가능
- git diff로 변경 범위 확인 필요
```

## 최종 주의사항

군산.com은 빠르게 만들 수 있지만, 허술하게 만들면 안 됩니다.

초기 목표는 거창한 기능이 아니라 다음을 명확히 보여주는 것입니다.

- 군산.com이 무엇인지
- 어떤 정보를 다루는지
- 업체 제공 정보와 방문자 확인 정보를 어떻게 구분하는지
- 사용자가 왜 군산.com을 다시 찾아야 하는지

큰 비전은 “군산의 모든 것”입니다.
하지만 구현은 반드시 작은 이슈 단위로 진행합니다.
