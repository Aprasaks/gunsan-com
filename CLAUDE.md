# Claude 작업 지침

## 역할

Claude는 군산.com 프로젝트의 개발 파트너입니다.

Claude는 항상 `AGENTS.md`의 모든 규칙을 우선적으로 따릅니다.

군산.com의 큰 브랜드 방향은 다음과 같습니다.

> 군산.com = 군산의 모든 것

군산.com은 단순한 지역 검색 사이트가 아니라, 군산의 생활 정보와 관광 정보를 한곳에 모으는 **군산 대표 정보 포털**을 지향합니다.

현재 프로젝트는 초기 placeholder 단계에서 벗어나, 사용자가 제공한 레퍼런스 이미지를 기준으로 **레퍼런스형 포털 홈**을 구축하는 단계입니다.

Claude는 작업할 때 다음 기준을 항상 기억합니다.

- 군산.com은 광고판이 아닙니다.
- 군산.com은 블로그가 아닙니다.
- 군산.com은 앱이 아닙니다.
- 군산.com은 군산 시민과 군산 여행자가 함께 쓰는 지역 정보 플랫폼입니다.
- 업체 제공 정보와 방문자 확인 정보는 반드시 구분합니다.
- 실제 데이터가 없는데 가짜 수치, 가짜 날짜, 가짜 실시간 상태를 만들지 않습니다.
- 메인 화면에 “준비 중”, “정리 예정”, “연결 예정” 문구를 반복하지 않습니다.
- 레퍼런스 이미지의 포털형 밀도를 우선합니다.

---

## 현재 작업 방향

현재 군산.com은 **레퍼런스형 포털 홈 구축 단계**입니다.

현재 단계에서 중요한 작업은 다음과 같습니다.

- 2단 포털형 Header
- 군산 현재 날씨 표시
- full-width 히어로 배너
- 카테고리 카드 UI
- 지금 많이 찾는 정보 chip/grid UI
- 군산 장소 둘러보기 preview
- 장소 카드에서 상세페이지로 이동하는 구조
- 오늘의 군산 영역
- 행사·축제 compact 영역
- 지도 preview 영역
- 동네별 보기 영역
- `/places` 장소 탐색 페이지
- `/places/[slug]` 장소 상세 페이지
- `/submit` 제보 페이지
- `/owners` 사장님 등록 안내 페이지

단, 큰 기능은 한 번에 붙이지 않습니다.

Supabase, 지도 SDK, 사진 업로드, 관리자 페이지, 로그인, 결제, 행사/관광 대량 API 연동은 사용자가 별도 작업으로 명시하지 않는 한 구현하지 않습니다.

## 반드시 따를 문서

작업 전 다음 문서를 읽고 따릅니다.

- `AGENTS.md`
- `docs/WORKFLOW.md`
- `docs/SKILL.md`
- `docs/DESIGN.md`
- `docs/GOAL.md`가 존재하는 경우 `docs/GOAL.md`

각 문서의 역할은 다음과 같습니다.

| 문서               | 역할                                              |
| ------------------ | ------------------------------------------------- |
| `AGENTS.md`        | 프로젝트 전체 작업 규칙                           |
| `docs/WORKFLOW.md` | 이슈·커밋·푸시·브랜치·AI 권한 기준                |
| `docs/SKILL.md`    | 타입 정의, 더미 데이터, 컴포넌트/페이지 구현 규칙 |
| `docs/DESIGN.md`   | 레퍼런스형 포털 홈 디자인 규칙                    |
| `docs/GOAL.md`     | 군산.com의 브랜드 목표와 제품 방향                |

`docs/GOAL.md`가 아직 없다면 `AGENTS.md`의 프로젝트 개요와 제품 방향을 기준으로 작업합니다.

문서끼리 충돌할 경우 우선순위는 다음과 같습니다.

1. 사용자의 현재 지시
2. `AGENTS.md`
3. `docs/WORKFLOW.md`
4. `docs/SKILL.md`
5. `docs/DESIGN.md`
6. `docs/GOAL.md`

단, 큰 방향이 바뀐 것 같으면 임의로 판단하지 말고 사용자에게 먼저 묻습니다.

## 작업 전 반드시 확인할 것

작업을 시작하기 전 반드시 다음을 확인합니다.

- 현재 작업이 어떤 작업명 또는 GitHub Issue에 해당하는지 확인합니다.
- 이번 작업에서 수정할 파일을 확인합니다.
- 이번 작업에서 절대 수정하면 안 되는 파일을 확인합니다.
- 관련 파일을 먼저 읽고 시작합니다.
- 요청 범위 밖 파일을 수정하지 않습니다.
- 한 작업에 여러 목적을 섞지 않습니다.
- 문서 작업과 코드 작업을 섞지 않습니다.
- UI 작업과 API 연결을 섞지 않습니다.
- 데이터 구조 변경과 디자인 변경을 섞지 않습니다.
- 사용자가 요청하지 않은 기능을 미리 구현하지 않습니다.
- 실제 데이터처럼 보이는 가짜 정보를 만들지 않습니다.
- 외부 사진, 리뷰, 지도 이미지를 무단으로 사용하지 않습니다.

작업 범위가 명확하지 않으면 먼저 사용자에게 묻습니다.

## 레퍼런스 이미지 기준

사용자가 제공한 레퍼런스 이미지는 메인 홈 디자인의 기준입니다.

Claude는 메인 홈 관련 작업에서 다음 방향을 우선합니다.

### Header

- 2단 포털형 Header
- 왼쪽 `군산.com` 로고
- 긴 캡슐형 검색창
- 오른쪽 날씨, 공지사항, 제보하기, 즐겨찾기, 사장님 등록
- 두 번째 줄 카테고리 내비게이션
- Header 안에 “준비 중” 문구를 반복 노출하지 않음

### Hero

- 박스형 카드가 아니라 full-width 파노라마 배너
- 텍스트와 floating card는 내부 max-width 컨테이너 안에 배치
- 이미지보다 검색과 핵심 문구가 먼저 읽히게 구성
- 관광 포스터처럼 과장하지 않음

### Category

- 작은 카드형 카테고리 진입 UI
- 부제목/설명 제거
- 카테고리명은 왼쪽 중앙
- 이미지는 오른쪽에 작게 배치
- 모든 카드 높이 통일
- “준비 중” 라벨 반복 금지

### PopularInfoSection

- 긴 설명 카드가 아니라 chip/grid UI
- 정보명만 짧게 표시
- 2열 compact grid 우선
- “실시간 인기”, “조회수”, “오늘 많이 본” 같은 표현 금지
- “준비 중”, “정리 예정”, “연결 예정” 문구 금지

### Place Preview

- 홈의 “군산 장소 둘러보기”는 preview 역할
- 카드 클릭은 모달보다 상세페이지 이동 우선
- 더 보기 버튼은 `/places`로 연결
- 각 장소는 고유 URL을 가져야 함

## API 작업 기준

API는 하나의 작업에서 하나만 연결합니다.

현재 우선 허용되는 API는 다음입니다.

- 군산 현재 날씨 API

API 작업 시 반드시 지킬 것:

- API 호출 로직은 컴포넌트 안에 길게 작성하지 않습니다.
- `lib/` 또는 `lib/api/` 아래로 분리합니다.
- 로딩 상태를 처리합니다.
- 실패 상태를 처리합니다.
- fallback을 둡니다.
- 실제 값이 없으면 임의 수치를 만들지 않습니다.
- API 작업과 UI 대규모 보정을 한 커밋에 섞지 않습니다.
- 지도 API, 관광 API, 행사 API는 별도 작업으로 분리합니다.

권장 파일:

```txt
lib/api/weather.ts
```

## 코드 작성 규칙

- TypeScript `any`를 사용하지 않습니다.
- 타입이 불명확하면 `unknown`을 사용하고 narrowing합니다.
- 타입은 가능한 한 `types/` 폴더에 분리합니다.
- 컴포넌트 내부에 임시 타입을 남발하지 않습니다.
- 업체 제공 정보와 방문자 확인 정보를 하나의 타입이나 컴포넌트 안에서 섞지 않습니다.
- 확인되지 않은 정보를 “최근 확인됨”처럼 긍정적으로 꾸미지 않습니다.
- 외부 사이트의 사진, 리뷰, 본문은 무단 복사하지 않습니다.
- 실제 기능이 없는데 동작하는 것처럼 보이게 만들지 않습니다.
- 컴포넌트가 너무 커지면 섹션 단위로 분리합니다.
- API 호출 코드는 컴포넌트 내부에 길게 쓰지 않습니다.
- 코드 변경 후 `npm run lint`와 `npm run build`를 확인합니다.

## 필드명 규칙

`docs/SKILL.md` 기준을 따릅니다.

- 동네/지역 필드는 `area`를 사용합니다.

- `neighborhood` 필드는 사용하지 않습니다.

- 검증 상태 목록은 `statuses`를 사용합니다.

- `verificationStatuses` 필드는 사용하지 않습니다.

- 업체 제공 정보는 `ownerInfo`를 사용합니다.

- 방문자 확인 정보는 `visitorInfo`를 사용합니다.

- 카테고리 표기는 `/`를 사용합니다.
  - `네일/뷰티`
  - `병원/약국`
  - `관광/여행`

- 카테고리 표기에 `·`를 사용하지 않습니다.

## 장소 정보 원칙

장소 정보는 군산.com의 신뢰 구조에서 가장 중요합니다.

Claude는 장소 관련 작업에서 다음을 지킵니다.

- 장소 카드는 상세페이지 이동을 우선합니다.
- 모달은 기본 구조로 사용하지 않습니다.
- 모달은 나중에 지도 핀 또는 빠른 미리보기 용도로만 검토합니다.
- `/places`는 탐색 페이지입니다.
- `/places/[slug]`는 장소 하나의 신뢰 정보 중심입니다.
- 업체 제공 정보와 방문자 확인 정보는 반드시 나눕니다.
- 확인되지 않은 정보는 확정적으로 표현하지 않습니다.
- 별점 중심 UI를 만들지 않습니다.
- 광고처럼 보이는 장소 추천 UI를 만들지 않습니다.

## 이미지 asset 기준

현재 단계에서는 직접 제작하거나 사용 권한이 확인된 이미지 asset을 사용할 수 있습니다.

허용:

- 사용자가 직접 제작한 이미지
- AI로 직접 생성한 이미지
- 직접 촬영한 이미지
- 사용 권한이 확인된 이미지
- 업체/방문자/관리자가 제공한 이미지
- 출처와 사용 조건이 확인된 공공기관 이미지

금지:

- 외부 사진 무단 다운로드
- 외부 리뷰 이미지 무단 사용
- 외부 지도 이미지 무단 사용
- 실제 군산 장소처럼 보이는 허위 AI 이미지
- 실제 상호, 실제 간판, 실제 건물을 허위로 생성하는 것
- 업체 제공 사진과 방문자 사진을 섞는 것

이미지 추가 작업에서는 다음을 확인합니다.

- 파일 경로
- 파일 용량
- 사용 위치
- alt 문구
- 사용 권한 또는 제작 방식

## 문구 작성 기준

군산.com의 문구는 짧고 명확해야 합니다.

허용 방향:

- 군산의 모든 것
- 군산에 산다면, 군산.com
- 군산에 여행 가고 싶다면, 군산.com
- 군산 생활과 여행 정보를 한곳에서 확인하세요
- 업체 제공 정보와 방문자 확인 정보를 나눠서 보여줍니다
- 정보가 달라졌다면 제보해주세요

주의:

- 광고처럼 과장하지 않습니다.
- 행정 문서처럼 딱딱하게 쓰지 않습니다.
- 실제 기능이 없는데 가능한 것처럼 쓰지 않습니다.
- 메인 화면에 “준비 중”, “정리 예정”, “연결 예정”을 반복하지 않습니다.
- 실제 검색량처럼 보이는 표현을 만들지 않습니다.
- “실시간 인기”, “조회수”, “오늘 많이 본” 같은 표현을 쓰지 않습니다.
- 실제 날짜, 행사 상태, 장소 수를 허위로 만들지 않습니다.

## 응답 언어

- 한국어로 응답합니다.
- 작업 보고도 한국어로 작성합니다.
- 커밋 메시지도 한국어로 제안합니다.
- 사용자가 질문하거나 방향을 고민하는 경우, 바로 구현하지 말고 먼저 의견을 정리합니다.
- 사용자가 “나랑 이야기 먼저 해”라고 말한 경우, 코드나 문서 수정 지시를 바로 만들지 말고 선택지를 비교합니다.

## 이슈커밋푸시 자동 진행

사용자가 다음과 같이 명시하면, 해당 작업에 한해 이슈 생성 → 검증 → 커밋 → 푸시를 순서대로 실행할 수 있습니다.

- `이슈커밋푸시 진행해`
- `이 작업 이슈 만들고 커밋하고 푸시까지 해줘`
- `issue commit push까지 진행해`

상세 순서, 중단 조건, 권한 기준은 `docs/WORKFLOW.md`의 **이슈커밋푸시 자동 진행 규칙**을 따릅니다.

단, 사용자가 명시하지 않으면 commit/push를 실행하지 않습니다.

## commit/push 권한 기준

Claude는 사용자의 명시적 지시 없이 다음을 실행하지 않습니다.

- `git commit`
- `git push`
- `git push --force`
- 배포 설정 변경
- 환경변수 변경
- 외부 서비스 연결

사용자가 `이슈커밋푸시 진행해`라고 명시한 경우에만 해당 작업 범위 안에서 commit/push까지 진행할 수 있습니다.

다음 작업에서 다시 commit/push가 필요하면 사용자가 다시 명시해야 합니다.

## 작업 완료 보고 형식

작업이 끝나면 반드시 아래 형식으로 보고합니다.

```txt
변경 파일
- ...

생성 파일
- ...

주요 변경
- ...

확인
- ...

수동 확인한 페이지
- ...

레퍼런스와 가까워진 점
- ...

아직 다른 점
- ...

TODO
- ...

제안 커밋 메시지
- 작업 내용 #이슈번호
```

코드 변경이 있는 경우 확인 항목에는 다음을 포함합니다.

```txt
확인
- npm run lint
- npm run build
```

문서만 수정한 경우에는 다음처럼 보고합니다.

```txt
확인
- 문서 작업이므로 npm run build는 생략 가능
- git status 확인
- git diff 확인
```

커밋 메시지의 `#번호`는 실제 GitHub Issue 번호를 사용합니다.

GitHub Issue 번호가 불명확하면 임의 번호를 붙이지 않습니다.

## 자동 진행 중단 조건

다음 상황에서는 작업 또는 commit/push를 중단하고 사용자에게 보고합니다.

- 작업 범위 밖 파일이 포함됨
- `.env` 또는 비밀 키 파일이 포함됨
- 의도하지 않은 lockfile 변경 발생
- 대규모 자동 포맷팅 발생
- `npm run lint` 실패
- `npm run build` 실패
- 이슈 번호가 불명확함
- 커밋 메시지에 실제 GitHub Issue 번호를 붙일 수 없음
- 사용자가 요청하지 않은 기능이 섞임
- 여러 API 연결이 한 작업에 섞임
- Supabase, 지도, 사진 업로드, 관리자 작업이 작은 UI 작업에 섞임
- 외부 사진, 리뷰, 지도 이미지 무단 사용이 포함됨
- TypeScript `any`가 포함됨
- 실제 데이터처럼 보이는 가짜 수치, 날짜, 실시간 상태가 포함됨
- 실패한 build 상태
- 사용자가 중단을 요청함

## 절대 하지 말 것

- TypeScript `any` 사용
- 업체 제공 정보와 방문자 확인 정보를 하나의 컴포넌트나 타입에 혼합
- 외부 사이트 콘텐츠 무단 복사
- 외부 데이터 무단 크롤링
- 외부 이미지 무단 사용
- 자유게시판 선제 구현
- 행사/축제 캘린더 선제 구현
- 지도 API와 관광 API와 행사 API를 한 번에 연결
- 사용자가 명시하지 않은 기능 추가
- 사용자 명시적 지시 없이 commit 실행
- 사용자 명시적 지시 없이 push 실행
- force push
- 작업 범위 밖의 파일 수정
- 이슈 없이 큰 기능 구현
- 한 커밋에 여러 기능 섞기
- 문서 작업과 코드 작업을 한 커밋에 섞기
- UI 작업과 API 연결을 한 커밋에 섞기
- 실패한 build 상태로 main push
- `neighborhood` 필드 사용
- `verificationStatuses` 필드 사용
- 카테고리 표기에 `·` 사용
- 메인 화면에 “준비 중”, “정리 예정” 문구 남발
- 실제 데이터처럼 보이는 가짜 수치 표시
- 실제 행사처럼 보이는 가짜 날짜 표시
- 실제 지도 기능처럼 보이는 허위 UI 구현
- 광고 사이트처럼 보이는 표현 사용
- 업체 홍보판처럼 보이는 구조로 변경
- 사용자가 제공한 레퍼런스 방향과 충돌하는 대규모 UI 변경
